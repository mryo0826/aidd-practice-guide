# 01. AI駆動開発（AIDD）全体戦略

## 前提思想

> **AIを使う開発でも、検査と適応の主体は人間である。** 本ガイドはアジャイル/スクラムの実践知を土台に、AIの提案を鵜呑みにせず人間が裁定者として判断する規律を、AIネイティブ開発に持ち込む（背景は `README.md` の「背景・設計思想」）。

## 1. Claude Code のインタラクションモード

### 各モードの特性と使い分け

| モード | 特性 | 最適な場面 |
|--------|------|-----------|
| **通常モード** | ファイル読み書き・コマンド実行が可能。自律的にタスクを遂行 | 実装、リファクタリング、バグ修正 |
| **plan mode** | 読み取り専用。編集・実行はせず設計や方針を協議する（`Shift+Tab` で切替） | アーキテクチャ検討、大規模変更の計画、調査・質問 |

### モード切り替えの判断基準

```
タスク開始
  ├─ 要件が曖昧 or 選択肢が多い → plan mode
  │    └─ 方針確定 → 通常モードへ切り替え
  ├─ 要件が明確 → 通常モード
  └─ 調査・質問のみ → plan mode（読み取りのみ）

実装中（通常モード）
  └─ 方針に迷いが生じた / 手戻りが連続した
       → STOP して plan mode へ戻る
       → 方針を再確定してから通常モードを再開する
```

> **STOP ルール:** 実装中に「この方針で合っているか？」と感じたら、そのまま進めずに plan mode へ切り替える。迷走したまま実装を続けるより、一度立ち止まって計画を修正する方がトータルコストが低い。

## 2. モデル選択の指針

### 用途別の推奨モデル（2026年7月時点）

| 用途 | 推奨モデル | 理由 | 具体的ユースケース |
|------|-----------|------|------------------|
| 概念整理・設計の壁打ち | Claude Opus 4.8 | 深い推論力、ニュアンスの理解 | 大規模リファクタリングの設計議論、ドキュメント体系の壁打ち、複数案の比較精査 |
| テンポの良い壁打ち | Claude Sonnet 5 | Opus級の知能 + 高速応答 | 方針が概ね固まった後の詳細化、.mdファイルの更新作業 |
| クロスレビュー・再検証 | 別系統の高性能モデル | Opusとは推論の癖が異なり、矛盾・欠落を検出しやすい | Opus成果物のセカンドオピニオン、仕様の整合性チェック |
| 大量コンテキスト分析 | 大容量コンテキスト特化モデル | 超大規模コンテキストウィンドウ | 大規模コードベースの俯瞰スキャン |
| 日常的なコーディング | Claude Haiku 4.5 | コスパと速度のバランス | 単発のバグ修正、定型的なコード変更、簡単な質問 |

### 既定モデルと切り替えの原則

- **既定は判断力の高いモデル**（設計・壁打ち・レビューは judgment 主体のため落とさない）
- **高速・低コストモデルへ落とすのは「大量の機械的編集フェーズ」に限定**（定型置換・一括生成・横断リファクタ等）。作業の性質を人が判断してから手動で切り替え、完了後に戻す
- **「Plan＝高性能／実行＝高速」を自動で紐づける運用は慎重に**。plan 境界に機械的に紐づくと、判断を含む実行まで無差別に格下げされる。切替は「作業が機械的か」で判断する
- 対象リポジトリが小規模な場合、モデル格下げで浮くTokenの絶対額は小さく、分割・格下げの主目的（節約）が効きにくい点も考慮する

### `既定` vs 明示指定の判断

- **既定モデルのまま**: 単純なコード修正、定型作業、素早い質問
- **明示指定**: 深い思考が必要な壁打ち、複雑な設計議論、長いコンテキストの維持

### 確度ラベル（認識論ラベル）

壁打ち・レビューでは、AIの発言を**重要度**（🔴🟡⚪）とは別軸の**確度（認識論ラベル）**で仕分けると、指摘の性質が明確になる。AI自身にこのラベルを付けさせる運用が有効。

| ラベル | 意味 | 扱い |
|--------|------|------|
| `[Fact]` | 検証可能な事実・論理／設計上の制約の誤り。正解が決まる領域 | 断定して修正する |
| `[Judgment]` | アーキテクチャ選択・優先順位・トレードオフなど主観に依存する領域 | 断定せず複数案・視点を提示する |
| `[Assumption]` | ユーザーが明示していない目的・制約・非機能要件・対象像 | 指摘の前に要確認 |
| `[To Be Verified]` | API変更など変化が速く、学習データが古い可能性が現実的に高い事実 | 確実なふりをせず必ずここに回し、裏取りする |

- **重要度ラベル（🔴🟡⚪）＝「どれだけ効くか」**、**確度ラベル（`[Fact]` 等）＝「どれだけ確かか」**。両者は直交する軸として併用できる
- 構造が確実で数値だけ不確かなら `[Fact]` +「(要数値確認)」と局所注記する。曖昧な事実を断定で創作しない
- `[To Be Verified]` は列挙で止めず裏取りし、確度が上がった項目は `[Fact]` 等へ格上げする

### クロスレビュー運用（実体験ベース）

同一の成果物を、初稿を書いたのとは**独立した視点**で再検証し、バイアスを相殺する手法。

> **本質は「別モデル」ではなく「独立性」:** 効果の大半は、**推論過程を見ていない目**で読み直すことから来る（推論過程を見ているレビュアーは結論を追認しやすい）。別ベンダーのモデルを用意できなくても、独立性は段階的に確保できる。

**独立性を確保する3段階（コスト順）:**

| 段階 | 手段 | コスト | 得られる効果 |
|------|------|--------|-------------|
| 1 | 新セッション＋`@`参照で対象だけ渡し、「前提を追認するな」と明示する | 無料 | 追認バイアスの除去（効果の大半） |
| 2 | レビュー用 Skill を当てる（観点・出力形式を定型化） | 無料 | 観点の抜け防止・再現性 |
| 3 | 別系統の高性能モデルでレビューさせる | 複数ベンダーのアカウントが必要 | 推論の癖の違いまで相殺（最も強い） |

**段階1だけでも成立する。** 段階3は「できれば」であり、クロスレビューの前提条件ではない。

**段階3（別系統モデル）を採る場合の役割分担:**

| ステップ | モデル | 役割 |
|---------|--------|------|
| 1. 初稿作成・壁打ち | Opus | 深い推論で方針を練る |
| 2. 俯瞰スキャン・再検証 | 別系統の高性能モデル | Opusとは異なる視点で矛盾・欠落・抽象性を検出 |
| 3. 精査・裁定 | Opus | レビューモデルの指摘を精査し、採用/修正/却下を判断 |

**注意:** 各モデルは自己能力を過大評価する傾向がある。モデルAが「自分が最適」と主張する推薦は、モデルBの視点で検証する。

**実証（本ガイド執筆での段階3適用時）:** 別系統モデルを使う場合、クロスレビュー役には、大量コンテキストの一括読み込みが得意なモデルよりも、Opusとは推論の癖が異なる別系統の高性能モデルの方が、成果物に対する再検証精度が高いことが判明した。大容量コンテキスト特化モデルは俯瞰スキャンには依然有効だが、クロスレビューの主役は別系統の高性能モデルに置くのがよい。

**クロスレビューの指摘の扱い方（実証済み）:**

| 指摘の種別 | 扱い |
|-----------|------|
| 明確な誤りや矛盾の指摘 | 採用（修正する） |
| 「より良い代替案」の提案 | 精査して採用/却下を判断 |
| モデル自身への誘導（「自分を使え」等） | 却下（自己バイアス） |
| 抽象度が上がりすぎる提案 | 一旦保留し実証後に判断 |

> クロスレビューは「AIの指摘を全部採用する」ものではない。人間が裁定者として機能することが前提。

> **採否を判断しうる出力には一意なIDを振る:** レビューの指摘に限らず、**人間が採用／却下を判断しうる出力単位すべて**（指摘・提案・意見・前言撤回・選択肢・要確認事項）に、安定したID（`F1`, `F2`…）と状態（採用／却下／保留／次へ送り）を与え、表形式で提示する。人間が『F3 却下』のように ID 単位で裁定・指示・書き戻しでき、長い散文を毎回引用し直す手間が消える。

> **`[Fact]` 適用範囲を「指摘」に狭めると規律が破綻する:** 本ガイドの公開前監査で、この規律の対象を「レビュー・監査の指摘」と定義したところ、AI が返す**提案・意見・前言撤回が規律の外に落ち**、散文へ逆戻りする事象が繰り返し発生した。狭い定義は抜け道を作る。**範囲は「採否対象すべて」で定義する**こと。テンプレートへの反映は `05_templates-and-patterns.md` セクション1.5。

**クロスレビューの実施手順:**

レビューを走らせる2パターン。**段階1ではモデルを切り替えず、新セッションを切るだけでよい**（下表の「新セッション + @参照」がそのまま段階1にあたる）。段階3では、ここでモデルも切り替える。

| 方式 | やり方 | いつ使うか |
|------|--------|-----------|
| 同一セッション内切替 | モデルを切り替え、レビュー指示テンプレートを投入 | 会話が短い（〜10ターン程度）とき |
| 新セッション + @参照 | 新セッションで対象mdを@参照し、背景1〜2文 + レビュー指示を投入 | 会話が長くなったとき、または重要な成果物 |

基本は同一セッション内切替で始め、「会話が長くなったな」と感じたら新セッションに切る。厳密なターン数閾値は不要。

**同一セッション内切替の注意点:**
- 文脈は会話履歴で100%引き継がれるため、引き継ぎテンプレートは不要。必要なのは「タスク切替宣言」（＝レビュー指示）
- 会話履歴が長いとToken効率が悪化しやすい（レビュー1回で会話全体が再処理される。プロンプトキャッシュで大半はキャッシュ読み出しになるが無料ではなく、レビューの出力生成コストも乗る）
- レビュアーが推論過程を見ているため、結論を追認するバイアスが生じうる。テンプレートに「前提から疑う」旨を明示する

**裁定後の書き戻し:** 裁定結果は対象Markdownか `docs/design-decisions.md` に1〜3行で書き戻す。同一セッション内では同期されるが、セッション終了後に判断経緯が消失するのを防ぐ。

> テンプレート本文は `05_templates-and-patterns.md` セクション1.5 を参照。

**プロンプト下書きバッファ（IME・日本語環境の協働パターン）:**

日本語ネイティブは、IMEの変換操作や複数行入力の途中で誤ってメッセージを送信しやすい。加えて、未整理の思考をそのままAIに渡すと精度が落ちる。対策として、入力前に一時的なMarkdownファイル（例: `temp.md`。決定ログ等の恒久ファイルとは別の一時ファイル）に指示内容を下書きし、変換・推敲を済ませてから流し込む運用が有効。

```
指示を temp.md に下書き（IME変換・推敲）
  → 内容を確認・整理
  → セッションへコピーして送信
  → temp.md は追跡しない（.gitignore 済み）
```

> このバッファは ①誤送信防止 ②未整理の思考を渡す前に構造化 の両方に効く、日本語環境での中核的な協働パターン。バッファ名は任意（`temp.md` / `prompt-draft.md` 等）。

> **下書きバッファは「公開しない」かつ「自動読込させない」の両方を掛ける:** この2つは別々の問題を解くため **and 条件で両方必要**。①`.gitignore` に入れて追跡・公開を止める（ローカルの読み込みは止まらない）。②Claude Code の permissions で Read を拒否し、自動読込・Token浪費・未確定メモの混入を止める（公開は止まらない）。Read拒否は `.claude/settings.json` の `deny` に `Read(./temp.md)` を追加する（本リポジトリでは設定済み）。

## 3. AI駆動開発のフロー

### 3.1 開発サイクル

```
1. 構想・壁打ち（plan mode / セッション）
   ↓
2. 設計確定（Markdown に書き出し）
   ↓
3. 実装（通常モード）
   ↓
4. レビュー・検証（通常モード）
   ↓
5. ナレッジ蓄積（Skills / CLAUDE.md 更新）
   ↓
   → 1 に戻る
```

### 3.2 壁打ちの進め方

> **核心原則:** 1つの長いセッションを続けない。Markdownを外部記憶として使い、トピック別にセッションを切る。

**Token コストの構造（プロンプトキャッシュ前提）:**

- `[Fact]` 各メッセージで全履歴が再送されるが、プロンプトキャッシュにより直前までの安定プレフィックスは**キャッシュ読み出し（入力の約0.1倍の単価）**で処理される。ゆえに「会話が長いほど1メッセージが加速度的に高騰する」わけではない（単価が上がるのではなく、扱う量が緩やかに増えるだけ）。
- `[Fact]` コストの主ドライバは会話の長さより次の2つ:
  - **出力（生成）量** — 出力は入力より高単価（Opus 級で約5倍）。丸ごと書き換え（Write）の多発が効くので、可能ならターゲット Edit を選ぶ。
  - **大きな一度きりの取り込み** — 巨大ファイルや参照ドキュメントの読み込み。必要な分だけ `@` 参照で渡す。
- `[Judgment]` それでもトピック別にセッションを切る運用は有効: 長い履歴のキャッシュ読み出し（0.1倍だが無料ではない）の累積と、大型コンテキストの持ち回り・再取り込み・再生成を避けられる。
- `[Fact]` **長い空白を挟む再開はキャッシュミスを生む。** キャッシュ TTL（1時間 / 既定5分）を超えて途切れると、次の1リクエストで累積プレフィックス全体を**書き込み（1時間 TTL で約2倍＝読み出しの約20倍の単価）で作り直す**。空白を跨ぐ長寿命セッションはこれを繰り返す。アイドル時間そのものは課金されないが、再開の最初の1発が高くなる。
- `[Fact]` **新セッションでも空白後の最初の1発は cold write になる。** キャッシュは共有されず、TTL を超えた再開は新旧どちらも cold。cold write 自体は空白を跨ぐ限り避けられない（回避は TTL 内の連続利用か事前ウォームのみ）。
- `[Judgment]` 差が出るのは cold write の**対象サイズ**だけ。巨大セッションの再開は累積プレフィックス全体（数十万トークン）を焼き直すが、小さく始めた新セッションは小さなプレフィックスで済む。「避ける」のでなく「**焼き直す量を小さくする**」のが正しい理解。
- `[Judgment]` 何をロードするかは人間に毎回選ばせない。**再開手順を `CLAUDE.md` に、着手タスクが触るファイル一覧を `BACKLOG.md` の各項目に符号化**し、AI がそれに従って最小ロードで再開する（`session-resume` Skill として横展開する候補）。
- `[To Be Verified]` プラン使用量メーターが cache write を read よりどう重く数えるかの正確な式は非公開。API 価格では cold write が高いのは確実で、この再開コストは実消費に効くと考えられる。

**推奨: トピック別セッション + Markdown外部記憶パターン:**
```
セッション A:「ツール選定の議論」
  → 確定内容を 02_*.md に書き込む → セッション A 終了
```

### 3.3 AI支援デバッグパターン

複雑な問題（インフラ設定、認証フロー、依存関係の問題等）をAIと協調して解決する際の一般パターン。

**基本パターン: 仮説→検証→棄却→次仮説**

```
問題を観察する（症状・エラーメッセージ・再現手順）
  ↓
仮説を生成する（AIが複数候補を提示。人間が優先順を決める）
  ↓
検証する（コマンド実行・ログ確認・設定確認。人間が実施）
  ↓
結果をAIに渡す
  ├─ 仮説が正しかった → 修正を実施
  └─ 仮説が誤りだった → 棄却してAIが次の仮説を生成
       └─ 繰り返す
```

**役割分担の原則:**

| 役割 | 人間 | AI |
|------|------|-----|
| 症状の観察・記録 | ✅ | 補助 |
| 仮説の生成 | 方向性の判断 | ✅ 複数候補を提示 |
| 検証の実施 | ✅（クレデンシャル・本番環境操作） | コマンド提案 |
| 結果の解釈 | 最終裁定 | ✅ 候補の絞り込み |
| 修正の実施 | 承認 | ✅ コード・設定変更 |

> **なぜ人間が検証を担うか:** クレデンシャルを含む操作・本番環境への影響・ブラウザ操作はAIに直接実行させない。「AIが仮説生成、人間が検証と裁定」の分担が安全かつ効率的。

**適用例（クラウドインフラの接続問題調査）:**

配信・フィルタリング・ネットワーク制御・認証といった複数の層をまたぐ接続障害は、単一の原因に絞り込みにくく、このパターンが効きやすい。
- AIが層ごとに仮説を並列提示する（配信設定の誤り、フィルタリングルールによる遮断、ポート制限、認証設定の不一致など）
- 人間がCLIで各層の設定を実際に確認し、仮説を検証・棄却する
- 数サイクル繰り返して根本原因に到達する

**モード選択との連携:**

```
デバッグの性質
  ├─ 仮説が不明・広い → plan mode（仮説の列挙・優先度付けから始める）
  ├─ 仮説が絞られている → 通常モード（検証コマンドの生成・実行）
  └─ ログ・コード読み込みのみ → plan mode（読み取りのみ）
```

### 3.5 サブエージェントの活用原則

Claude Code のサブエージェント（Task tool）は、メインの文脈を汚さずに並列で探索・調査・分析を行うための手段。

**使いどころ:**

| 場面 | サブエージェントを使う理由 |
|------|--------------------------|
| コードベースの広範囲探索 | メイン文脈にノイズを入れず、結果だけ受け取る |
| 複数ファイルの並列調査 | 直列よりもToken効率・速度が良い |
| 独立したサブタスクの実行 | メインの推論過程を中断しない |

**原則:**
- サブエージェントは「調べてきて結果だけ教えて」の委任モデルで使う
- メインエージェントが方針決定・統合を担い、サブエージェントは探索・収集に徹する
- サブエージェントへの指示はメイン会話の文脈を含まないため、自己完結した指示を渡す必要がある

> **`[Fact]` サブエージェントは Skills の中身を自動では受け取らない（公式ドキュメント確認済み・2026-07-18 訂正）:** サブエージェントは親の `CLAUDE.md` は受け取るが、**プリロード済み Skill の中身は受け取らない**。カスタムサブエージェント（`.claude/agents/`）の定義にある `skills` フィールドに列挙した Skill だけが、起動時にコンテキストへ**中身ごと注入**される（説明文だけでなく全文）。
>
> ただし `skills` が制御するのは**「起動時にプリロードするか」であって「利用できるか」ではない**。列挙しなかった Skill も、サブエージェントが `Skill` ツールを持っていれば発見・呼び出しできる。つまりこのフィールドは**コンテキスト最適化のスイッチ**であり、アクセス制限ではない。
>
> したがって **Skill の中身が確実に効いている必要がある委任では、`skills` に明示列挙する**（呼び出し任せにすると、サブエージェントが呼ばない可能性が残る）。出典: [Claude Code Docs / Sub-agents](https://code.claude.com/docs/en/sub-agents)・[Agent SDK / Subagents](https://code.claude.com/docs/en/agent-sdk/subagents)。
>
> **`[Judgment]` この訂正自体が X1（`[Assumption]` 再点検）の成果である。** 前セッションでも公式ドキュメントで裏取りして `[Fact]` を付けたが、「プリロードされない」を「アクセスできない」と読み違えていた。**裏取り済みの `[Fact]` でも、出典の主張と本文の主張がズレることがある**——確度ラベルは「出典を見た」ことの証明であって、「正しく読めた」ことの証明ではない。

### 3.6 セッションをスプリントとして扱う（レトロで検査と適応）

AI協働のセッションを、アジャイル開発の**スプリント**（区切られた一定期間の作業単位）に見立てて回す。セッション終了時（または節目）に**レトロスペクティブ**（その期間の進め方自体を振り返る場。以下「レトロ」）を行い、人間側の「検査と適応」を明示的にループさせる。標準の KPT（Keep＝続けること／Problem＝問題／Try＝次に試すこと）に、「前回 Action の検証」「事実の棚卸し」「現実的な Action へのコミット」を足した形式が有効。

> **この節はアジャイルの前提知識を要求しない:** 以下の手順は、アジャイル／スクラムを知らなくてもそのまま実行できる形で書いてある。要するに「**やり方そのものを定期的に点検し、直すと決めたことが本当に直ったかを次回の冒頭で確かめる**」という運用ループであり、用語はその手順に付いた名前にすぎない。出自はスクラムのレトロスペクティブ（本ガイド著者がスクラムマスターとして実践の中で組み立てた形式）だが、価値は出自ではなく後述の失敗モードを塞げる点にある。

| フェーズ | 問い |
|---------|------|
| **Check Action** | 前スプリント（前セッション）で決めた Action はやれたか。やれた／やれなかった、で結果はどうか |
| **DO** | このスプリントで実際に何をやったか（判断の前に事実を棚卸しする） |
| **Keep / Problem** | 続けること ／ 問題だったこと |
| **Try** | 次に試したいこと。**各 Try に「処理」（Action へ昇格／実施済／見送り／人間側）を必ず付け、行き先が宙に浮く Try を残さない** |
| **Action** | Try から、次スプリントで現実的に実践できる数個を選んでコミットする。**各 Action に由来する Try の ID を明記する** |

> **なぜ Check Action を冒頭に置くか:** 多くの KPT は前回の改善合意を検証せず、Try が願望のまま蒸発する。冒頭で前回 Action の結果を検査することでループが閉じ、改善が定着する。Check Action は人間の記憶に頼らず、再開手順に組み込んで機械的に実行する（`CLAUDE.md`「再開手順」）。**これはツールの性能では塞げない穴である**——どれだけ強力なモデルを使っても、「次はこうする」と決めた合意を次回に検査する主体がいなければ改善は定着しない。道具（HOW）の改善と、進め方を検査するループは別の層にあり、後者を持たないチームは同じ問題を繰り返す。

> **`[Fact]` Try→Action のトレーサビリティを省くと認知負荷が跳ね上がる:** 各項目に ID を振っても、**Try と Action の対応を示さなければ**、読み手は「どの Try がどの Action になったか」を頭の中で再構成させられる（疲労時ほど負担が大きい）。**Try 表に「処理」列、Action 表に「由来」列**を持たせ、対応を目で追えるようにする。本ガイドのレトロ実施時に実際に発生した問題であり、ID を振るだけでは不十分で**ID 同士を繋ぐ**ことまでが要件。

> **AIとの相性:** レトロ結果（特に Action）は Markdown ／ 決定ログ ／ エージェントのメモリに書き戻す。次セッションの冒頭で AI に前回 Action を Check Action させれば、人間の判断責任を保ったまま検査と適応が継続する（役割分担は `README.md`・`CLAUDE.md` 参照）。

#### セッション終了の判断（出口の分担）

Check Action によって**入口**（再開手順）は仕組み化された。一方で**出口**——いつセッションを閉じ、いつレトロを回すか——を決める仕組みがなければ、区切りは人間の感覚値に委ねられたままになる。ここは分担を分ける。

| 誰が | 何を担うか |
|------|-----------|
| **AI** | **区切り候補を申告する。** 下記の機械的な材料が揃ったら、作業を続ける前に「区切り候補です」と申告する |
| **人間** | **終了／レトロ／続行を裁定する。** 疲労・残り時間・集中力は人間にしか読めない |

**AI が申告する根拠（機械的に判定できるものだけ）:**

1. **タスクが1つ閉じた** — 着手していた項目が完了し、次の項目に移る前である
2. **`docs/design-decisions.md`・`BACKLOG.md` への書き戻しが未了** — このまま終了すると次回の再開で読み直す量が増える（§3.2 の cold write）
3. **文脈が肥大している** — 会話が長くなり、参照ファイルが積み上がっている

> **なぜ終了判断を人間に残すのか:** **AI に検知できない信号があるため。** 本ガイドの執筆セッションで、著者は「疲れたので自分のミスが頻発しそう」と判断して終了を促した。これは AI 側からは観測できない。逆に、上記3点のうち **② 書き戻し未了は人間の感覚では検知できず、AI 側が検知できる領域**である。互いに見えないものを補い合う分担になっている。
>
> この線引きは「初稿を書いたモデルを唯一の裁定者にしない」（§2）と同じ構造である。**AI にできるのは判断の肩代わりではなく、判断材料の提供まで。**

> **`[Judgment]` 申告は提案であって停止ではない:** AI が申告したら人間が答えるまで作業を止める、という運用にはしない。止めると申告そのものが煩わしくなり、やがて無視されるようになる。申告は「今なら切りやすい」という情報提供であり、裁定がなければ作業は続く。

## 5. 既存コードベースへのAI駆動TDD

すでに動いているコードベースに機能変更・仕様変更を加える際の、AI支援TDDパターン。
新規開発（ゼロから作る場合）と異なり、**既存動作の保護**と**意図的な変更**を分離することが核心となる。

### 5.1 基本フロー

```
Phase 0: 特性テスト（Characterization Tests）
  ↓ 現行の動作をテストで先にロック（壊れた動作も含めて「現行仕様」として記録）
Phase 1: Red — 意図的な失敗
  ↓ 新しい仕様に対応するテストを書く → 意図的にテストを失敗させる
Phase 2: Green — 最小実装
  ↓ テストが通る最小の実装を行う
Phase 3: Refactor — 整理
  ↓ 既存テストがすべて通ることを確認しながらコードを整理
```

### 5.2 既存バグの扱い方

既存コードベースでは、現行の動作の中に「バグ」と「仕様」が混在している。修正前に分類する。

| 分類 | 説明 | Phase 0の扱い |
|------|------|---------------|
| **現行仕様としてロックするもの** | 今回変更しない既存動作（バグでも）。テストで固定してから進む | 特性テストに含める |
| **今回直すもの** | このタスクで明示的に修正するバグ・仕様変更 | Red テストとして先に書く |

> **なぜ分類が重要か:** 「既存バグも含めて全部直す」とスコープが無限に広がる。今回直すものを明示し、それ以外はロックすることで、AIへの指示も絞り込める。

### 5.3 AIとの協調ポイント

| タスク | 人間の役割 | AIの役割 |
|--------|-----------|---------|
| Phase 0: 特性テスト作成 | テスト対象の範囲を決める | 現行コードを読んで特性テストの雛形を生成 |
| Phase 1: Red テスト | 新仕様の意図を確定する | テストコードを生成・修正 |
| Phase 2: Green 実装 | レビューと承認 | 最小実装を提案・実行 |
| Phase 3: Refactor | 変更範囲と方針を決める | リファクタリングを実施しテストを再実行 |

### 5.4 適用例（入力検証ルールの変更）

**タスク:** 既存コードベースの入力値の検証ルールを変更する

**背景:** 検証ルールの定義と、それを参照する表示側が分かれており、さらに複数の箇所が同じルールに依存している——という構成はよくある。既存動作を壊さずに変更するには順序が要る。

**フロー:**
1. **Phase 0（特性テスト）:** 現行の検証ロジックに対するテストを先に書いて既存動作を固定
2. **Phase 1（Red）:** 新しい検証ルールに対するテストを追加 → 意図的に失敗させる
3. **Phase 2（Green）:** 正規表現定義と表示コンポーネントを最小限の変更で更新
4. **Phase 3（Refactor）:** 関連コンポーネントの整合性確認とクリーンアップ

**得られた知見:**
- 特性テストがあることで、AIが「どこまで変えてよいか」を明確に判断できた
- 既存動作のロックと新仕様の変更を分離したことで、AIへの指示がシンプルになった
- タスク固有の短命コンテキストファイルにPhaseごとの制約（変更してよいファイル・してはいけないファイル）を記載することが有効
