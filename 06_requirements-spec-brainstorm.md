# 06. 要件・仕様（WHAT）の壁打ち・レビュー方法論

> **このファイルの責務:** 開発の前段「何を作るか（WHAT）」を固める壁打ち・レビューの**方法論**。01〜05（どう作るか＝HOW）を補完する別レイヤー。
> プロダクト固有の要件そのもの（対象機能・数値・制約）は本ファイルに書かず、各プロジェクトの仕様ドキュメント／context ファイルに置く（本ファイルは再利用可能な方法論のみ）。

## 前提思想

> **仕様は How（実装方法）ではなく What（満たすべき状態）で固定する。確定した What には必ず受入基準（AC: Acceptance Criteria）を添える。**
>
> 要件・仕様フェーズの目的は「作るものの状態」を検証可能な形で確定させること。実装戦略（01〜05）に入る前に、What が falsify 可能な粒度で固まっているほど手戻りが減る。
>
> この WHAT レイヤーは、テクニカルPdM として要件・仕様まで射程を広げた拡張であり、アジャイルの「正しいものを正しく作る」の前半（正しいものを見極める）を担う。要件・仕様が曖昧なまま開発が進み全体像が見えなくなる実プロジェクトの経験から、「何を作るか」を検査可能な形で固める必要を痛感して切り出したのが本章である（背景は `README.md` の「背景・設計思想」）。

---

## 1. 位置づけ（WHAT と HOW の分離）

| レイヤー | 対象 | 該当 |
|----------|------|------|
| **WHAT** | 何を作るか（要件・仕様の定義・レビュー） | 本ファイル（06） |
| **HOW** | どう作るか（開発戦略・ツール・Skills・実装） | 01〜05 |

- この方法論は **methodology（再利用可能）** なので Skill 化して横断利用する。プロジェクト固有の要件 **content** は仕様ファイル／`CLAUDE.md` に置き、方法論と混ぜない（分離原則は `03_agent-skills-knowhow.md` セクション2 / `04_spec-kit-integration.md` セクション1.1）。
- Spec Kit を使う場合、本方法論は Specify（What/Why を定義する）フェーズの思考法にあたる（`04_spec-kit-integration.md` 参照）。

## 2. モードと基本スタンス

| モード | 発動 | 振る舞い |
|--------|------|---------|
| **壁打ち（既定）** | 対象の有無に関わらずここから始める | 設計の穴を質問と指摘で突いて固める。整形フォーマット（🔴🟡💡🔬）は使わず対話形式。固まった論点は『確定』と明示して攻撃対象から外し、未解決の弱点に火力を集中する |
| **レビュー** | 「レビューして」と明示宣言された時のみ | 固めた設計をセクション7の Output Format で構造化レビューする |

**スタンス:**
- **媚びない reviewer**。攻撃性・厳格さは口調ではなく **指摘の中身（content）** で出す。文体はフランクな丁寧語でよい。
- **対象が不十分ならフォーマットを埋めない**。「レビュー対象が不足しています。最低限必要な情報は［確定した設計 / 制約・非機能要件 / 未決の論点］です。」と一文だけ返す。
- **創作・推測での補完（Hallucination）をしない**。埋めるための想像で穴を塞がない。

## 3. 確度ラベル

要件・仕様の壁打ち・レビューでは、`01_ai-driven-dev-strategy.md` セクション2「確度ラベル（認識論ラベル）」をそのまま用いる（`[Fact]` / `[Judgment]` / `[Assumption]` / `[To Be Verified]`）。特に要件フェーズでは、ユーザーが明示していない目的・非機能要件・対象像は `[Assumption]` として**指摘の前に要確認**とするのが重要。

---

## 4. 行動規範

1. **褒めるだけの文言を出さない。** ただし「妥当なので攻撃対象から除外した」という判断の明示は可。
2. **古い前提を疑う。** 学習時点より後に、外部 API/クラウドの仕様・制限（レートリミット、廃止・非推奨）やモデルのライフサイクルが変わった可能性を常に疑う。ネットワーク遅延・コールドスタートも含め、物理的に破綻した箇所を突く。確実なふりをしない。
3. **正常系「動くこと」の礼賛に出力を使わない。** 「AI の誤認識」「ネットワーク瞬断」「同一リクエストの連投（冪等性バグ）」「認証トークンの失効」でシステムがどう死ぬかを主眼にする。正常系の設計自体に欠陥があればそれも指摘する。
4. **ハードな納期があれば欠陥指摘で終わらせない。** 工期内に間に合わないリスクのある重い要件・仕様は【Out of Scope】として積極的に切り捨て提案し、何を削れば死守ラインを守れるかまで踏み込む。具体的な納期・残工期はプロジェクトの context ファイルを参照する。

## 5. What > How 規律と受入基準（AC）

### 5.1 What で固定する

仕様は How（実装方法）ではなく What（満たすべき状態）で固定する。「このロール／この画面でどの状態が正か」の定義に火力を集中し、How に依存した曖昧な仕様は What へ巻き直すよう突く。

### 5.2 受入基準（AC）の必須化

論点を『確定』と宣言する際は、その What に対する **受入基準（Acceptance Criteria）** を必ず添える。

| AC の要件 | 内容 |
|-----------|------|
| **How を含めない** | 実装手段ではなく「満たすべき状態の検証条件」で書く |
| **観測可能** | 受け入れ検査で発注側／受け手がそのまま ○×判定できる具体度 |
| **強要しない範囲** | 未決論点には AC を強要しない（確定した What にのみ添える） |

> **なぜ AC が要るか:** AC は新概念ではなく「仕様は How でなく What で固定する」既存 doctrine の成果物形。確定要件が「もっともらしい散文」のままだと falsify（反証）できない。検証可能条件に落とすことでこの穴を塞ぐ。受け手が仕様駆動・テスト駆動に不慣れな場合ほど、AC を受け入れ検査にアンカーする価値が高い。

## 6. レビューの着眼点

設計を見るときは特に次を突く（思考手順は内的推論に委ね、観点のみ示す）。

- 前提と正常系の構造に、致命的な勘違い・古い仕様が紛れていないか。
- 関係する外部 API/クラウドの制限事項に抵触していないか（各々を確度ラベル付きで判定）。
- 自分の指摘に「この構成でも耐えるのでは？」と一度反論し、防げない本物の Showstopper だけを残す。
- 仕様が How ではなく What で固定されているか。
- 確定した What に AC が紐づいているか。AC が ①How を含まず満たすべき状態の検証条件で書かれているか ②受け手が受け入れ検査でそのまま ○×判定できる具体度か。抽象的すぎ・専門用語で相手が検証できない AC は突く。

## 7. レビュー Output Format（レビュー時のみ）

```
■ 🔴 致命的（今すぐスクラップ＆ビルドすべき設計欠陥）
■ 🟡 懸念（本番運用・例外系・仕様変更で障害になる点）
■ 💡 改善案 / 代替アーキテクチャ案
■ 🔬 要検証リスト（最大5件。各項目「何を」「どの公式ドキュメントで（可能なら具体的な URL）」をセットで）
```

- 各指摘には一意なID（`F1`, `F2`…）を付け、受け手が ID 単位で採否・対応を指示できるようにする。指摘に限らず、提案・意見・選択肢など**採否を判断しうる出力すべて**が対象（`01_ai-driven-dev-strategy.md` セクション2「採否を判断しうる出力には一意なIDを振る」）。
- 確定した各 What に AC が紐づいているかも確認し、欠落・How混入・検収不能な抽象 AC は 🔴/🟡 として扱う。
- 💡 は「改善案／代替アーキテクチャ案」（設計レベルの選択肢提示）。要件・仕様レビューでは実装コードそのものは求めない（役割スコープが WHAT のため）。

---

## 8. 運用規律

| 規律 | 内容 |
|------|------|
| **要検証リストの運用** | 🔬 / `[To Be Verified]` は最大5件。多ければ重要度で絞り、効かない項目はノイズとして切る。列挙で止めず裏取りし、確度が上がった項目は `[Fact]` 等へ格上げ、一次ソースに当たれないものだけ 🔬 に残す。確認した URL を併記する |
| **完成文の早出し禁止** | 検討・修正提案の段階では、理由と提案内容（箇条書き）のみを出す。清書（仕様ドキュメント上書き用の完成文章）は、明示指示があるまで出力しない |
| **確定物の永続化** | 確定した要件・仕様は仕様ドキュメント（git 追跡の正本）に残し、必要に応じてチーム共有基盤へ公開する。壁打ち途中の未確定論点を勝手に公開しない |
| **文脈と方法論の分離** | プロジェクト文脈（確定前提・確定 What・未決論点）は context ファイルを source of truth とし、方法論（本章）と役割を分離する。再開時は context ファイルと git ログから復元する |

## 9. Skill 化のヒント

本方法論は Skill（`SKILL.md`）として持つと、要件・仕様の壁打ちが必要なプロジェクトで横断的に再利用できる。

- **description 例:** 「要件・仕様の壁打ち＆レビュー。甘い前提・抜けた非機能要件・外部API/クラウド制約の見落としを、PdM 兼アーキテクト視点で媚びずに突く。要件定義・仕様策定・設計レビュー時に使用。」
- プロジェクト固有の要件・数値・納期は Skill に埋め込まず、context ファイル側に置く（`03_agent-skills-knowhow.md` セクション2 の methodology/content 分離）。
- Skill 設計の一般則は `03_agent-skills-knowhow.md`、確度ラベルは `01_ai-driven-dev-strategy.md` セクション2、ドキュメント記述規約は `02_mcp-and-tools-ecosystem.md` セクション6 を参照。
