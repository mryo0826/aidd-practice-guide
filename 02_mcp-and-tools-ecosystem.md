# 02. MCP・ツールエコシステム 設定・運用ノウハウ

## 前提思想

> **MCPは万能ではない。内蔵ツール・CLIで済むものはそちらを使う。**
>
> CLI（シェルコマンド）はコマンド文字列+標準出力のみで完結し、追加の常駐プロセスも環境依存の設定も要らない。
> Agent Skillsと組み合わせれば、CLIの使い方をAIに教えつつ、必要な時だけ知識を読み込ませられる。

> **⚠️ この章の根拠は2026年7月に更新された（重要）:** 本章は当初「**MCPのツール定義はコンテキストに常時ロードされるため、導入するほどToken消費が増大する**」を主根拠にしていた。**この部分は現在の Claude Code では既定で成立しない。**
>
> `[Fact]` Claude Code は **tool search を既定で有効**にしており、ツール定義は遅延ロードされる。セッション開始時にコンテキストへ入るのは**ツール名とサーバー指示のみ**。公式ドキュメントは「adding more MCP servers has minimal impact on your context window」と明記している。出典: [Claude Code Docs / MCP](https://code.claude.com/docs/en/mcp)
>
> **ただし結論は変わらない。主コストが別の軸にあることが分かっただけである。** 外部の報告では **Playwright MCP のブラウザ操作が1操作あたり 5,000〜10,000 トークン**を消費し、CLI + HTTP API 方式への置き換えで **約90%削減**できたとされる（§1.1 の引用を参照）。これは**呼び出しごとのレスポンス量**によるもので、tool search では解消しない。
>
> 加えて、Token以外の理由も CLI 優先を支持する:
> - **既定は普遍ではない** — `ENABLE_TOOL_SEARCH=false`、`tool_reference` 非対応モデル、一部のホスティング環境では**今も全ツール定義が先読みされる**。チームへ配布する設定ほどこの差に晒される
> - **運用コスト** — 常駐プロセス・per-machine な絶対パス設定・承認フロー・接続失敗時の切り分け・バージョン追従が増える
> - **代替可能性** — 内蔵ツールやCLIで同じことができるなら、依存を1つ増やす利得がない
>
> `[Judgment]` **この改訂の教訓:** 当初の主張は「結論は正しいが、挙げた根拠が後に偽になった」型の誤りだった。**結論が生き延びたのは、レスポンス量という別の根拠が実在したからで、論理が正しかったからではない。** 変化の速い事実に依存する主張は `[To Be Verified]` を付けて検査対象に残す（`01` §2）。

## 1. MCP vs CLI：判断フレームワーク

### 1.1 Token消費の構造的違い

MCPのToken消費には**独立した2つの軸**がある。混同すると判断を誤る。

| 軸 | 内容 | 現状 |
|----|------|------|
| **① ベースライン** | ツール定義がコンテキストに常駐する分 | **既定では概ね解消**（tool search により遅延ロード） |
| **② 呼び出しごとのレスポンス量** | 1回の操作で返る構造化データの量 | **未解決。MCPの主コストはこちら** |

```
MCP（tool search 有効＝Claude Code の既定）:
  ① ツール名+サーバー指示(常時) + 使ったツールの定義(必要時)
  ② 呼び出し引数 + 構造化レスポンス ← ここが大きい

MCP（tool search 無効の環境）:
  ① 全ツール定義(常時) → ツール数に比例してベースラインが増える
  ② 同上

CLI:
  ① なし（ベースライン増加なし）
  ② コマンド文字列 + stdout/stderr ← 必要な情報だけ返せる

Agent Skills + CLI:
  ① なし。Skillは関連時のみ読み込まれる
  ② 同上
```

> **`[Fact]` ① について — tool search の挙動:** 既定（環境変数未設定）では全MCPツール定義が遅延ロードされ、セッション開始時に入るのはツール名とサーバー指示のみ。`ENABLE_TOOL_SEARCH` で変更でき、`auto` はコンテキストウィンドウの10%に収まる分だけ先読み、`false` は全先読み（旧来の挙動）。`tool_reference` 対応モデル（Claude Sonnet 4.5 / Haiku 4.5 / Opus 4.5 以降）が必要。出典: [Claude Code Docs / MCP](https://code.claude.com/docs/en/mcp)

> **`[Fact]` ② について — MCPレスポンスは大きくなりやすい:** Claude Code は **MCP出力が10,000トークンを超えると警告し、既定で25,000トークンに制限する**（`MAX_MCP_OUTPUT_TOKENS` で変更可）。この制限が設けられていること自体が、MCPレスポンスが桁違いに膨らみうる実態を示している。出典: [Claude Code Docs / MCP](https://code.claude.com/docs/en/mcp)

> **📝 引用（Token消費 約90%削減の報告）:** `[Fact]` ブラウザ操作タスクにおいて **Playwright MCP は1操作あたり 5,000〜10,000 トークン**を消費するとの報告がある（操作ごとにページ状態の構造化スナップショットが返るため）。次の構成へ置き換えることで **Token消費を約90%削減**（複数タスクの比較。ログイン＋2段階認証の例では93.8%）、複数回の操作を含むタスクでは**10分の1**になったと報告されている。
>
> 出典: [CLIでもできた！PlaywrightMCPと同じ動き＋トークン90%削減](https://qiita.com/Aochan0604/items/38b4a6359662234245bb)（@Aochan0604、2026-02-11）
>
> **⚠️ これは著者自身の実測ではない。** 外部記事の報告を引用している。数値の再現性は原典の条件に依存する。
>
> ```
> Claude Code (Bash) → CLIスクリプト (curl) → HTTP APIサーバー → Playwright
> ```
>
> **効いた理由:** ブラウザ操作の実体をAPIサーバー側に閉じ込め、モデルには**必要な結果だけ**を stdout で返した。MCP は汎用性のためにページ状態を丸ごと構造化して返すが、タスクが必要としていたのはその一部だけだった。
>
> **一般化すると:** `[Judgment]` MCPのコストは「何回呼ぶか × 1回に返る量」で決まる。**高頻度・大レスポンスの操作（ブラウザ・大規模スキーマ探索など）ほどCLI側に寄せる価値が高い。** 逆に低頻度・小レスポンスならMCPの利便性が勝つ。§1.2 の判断マトリクスで「呼び出し頻度」を軸に入れているのはこのため。
>
> **この報告は tool search 導入後も有効である。** tool search が解決したのは軸①（定義の常駐）だけで、軸②（呼び出しごとのレスポンス量）には効かない。

### 1.2 MCP vs CLI 判断マトリクス

| 判断軸 | MCP を選ぶ | CLI を選ぶ |
|--------|-----------|-----------|
| **状態管理** | セッション跨ぎの状態保持が必要 | ステートレスで済む |
| **入出力** | 構造化データの双方向通信が必須 | テキスト入出力で十分 |
| **代替手段** | CLI相当が存在しない | 成熟したCLIツールが存在する |
| **呼び出し頻度** | 低頻度・高複雑度 | 高頻度・低複雑度 |
| **エラー処理** | プロトコル側の構造化エラーが必要 | exit code + stderr で十分 |

### 1.3 判断フロー

```
そのツールにCLI版はあるか？
  ├─ YES → CLIで要件を満たせるか？
  │    ├─ YES → CLI + Agent Skill で運用 ✅（最もToken効率が良い）
  │    └─ NO  → 状態管理 or 構造化I/Oが必要？
  │         ├─ YES → MCP を導入
  │         └─ NO  → CLI + スクリプトで補完
  └─ NO  → MCP固有の機能か？
       ├─ YES → MCP を導入（Serena等）
       └─ NO  → 汎用Shell操作で代替検討
```

> **📝 判断フローの適用例:**
>
> 入力検証ルール変更タスクでの使い分けトリガー適用:
> - **タスク特性**: 要件が明確、対象ファイルが少数、新機能ではなく簡易な仕様変更
> - **選択**: 「仕様Markdown + Skills」パターン（Spec Kit フルフローは不要）
> - **根拠**: 使い分けトリガー（`04_spec-kit-integration.md` セクション2）の「軽量変更（仕様を先に固定したい変更）→ 仕様Markdown + Skills」に該当
> - **Serena MCP**: コード構造の探索に使用（検証関連ファイルの依存関係把握）。実装変更自体は通常モード + CLI で実施

> **📝 AI協調デバッグでのCLI使い分け:**
>
> クラウドインフラ（配信・フィルタリング・ネットワーク制御・認証）の接続問題調査:
> - **調査フロー**: AIが仮説を複数提示 → 人間がCLIでリソース設定を確認 → 結果をAIに渡して次仮説へ
> - **CLIを選んだ理由**: 各層のリソース設定の参照はクラウドプロバイダのCLIで完結でき、追加のツールを挟む必要がない
> - **MCPを使わなかった理由**: CLI で十分かつ Token 効率が高い。セキュリティ上もCLI経由の方が制御しやすい
> - **Serena MCP の用途**: コード側の構造調査に限定。インフラリソースの直接確認はCLIを使用
> - **教訓**: インフラ調査では「仮説生成はAI、検証は人間がCLI」の分担が最も効率的

> **📝 判断フロー実例：LSP（Language Server Protocol）ベースMCP（serena等）の見送り:**
>
> Markdown中心でコードを持たないリポジトリ（ドキュメント／プロンプト集など）での判断:
> - **判断フロー適用**: 「CLI版はあるか？」以前に、そもそもMCP固有機能（serenaのLSPベースのシンボル検索・参照追跡・セマンティック編集）の**対象（シンボル・参照）が存在しない**
> - **選択**: LSPベースMCPを導入しない。内蔵の Read/Grep/Glob/Edit で過不足ない
> - **根拠**: MCPを足すとプロセス・per-machine依存・ツール定義のTokenだけが増え、「Token削減優先」と逆行する
> - **教訓**: MCPの価値は「対象データ構造」に依存する。実コード（ビルド/検証スクリプト等）が入った段階で再検討すればよい

## 2. 主要ツール別：推奨構成

### 凡例
- ✅ 推奨  ⚠️ 条件付き  ❌ 非推奨（Token効率の観点）

### 2.1 ブラウザ操作 / E2Eテスト

| 方式 | Token効率 | 推奨 | 備考 |
|------|-----------|------|------|
| `playwright` CLI / `npx playwright` | 高 | ✅ | テスト実行・スクリーンショット等は CLI で十分 |
| HTTP API + CLIスクリプト（自作） | 高 | ✅ 多操作時 | ブラウザ操作をAPIサーバー側に閉じ込め、必要な結果だけを返す（§1.1 の引用で約90%削減の報告） |
| Playwright MCP | 低 | ⚠️ | **1操作あたり 5,000〜10,000 トークン**（§1.1 の引用）。操作回数が少なく、対話的な探索が必要な場合に限る |

**推奨構成:**
```
基本: playwright CLI（テスト実行、スクリーンショット取得）
補助: Agent Skill で playwright CLI の使い方をAIに教える
多操作: HTTP API + CLIスクリプト方式（Claude Code → curl → APIサーバー → Playwright）
例外: 対話的な探索が必要で、かつ操作回数が少ない場合のみ Playwright MCP
```

> **判断基準は操作回数:** 1〜2操作で終わるなら MCP の手軽さが勝つ。**同じタスク内で何度もブラウザを操作するなら、1操作 5,000〜10,000 トークンが累積するため CLI 側へ寄せる。**

### 2.2 仕様駆動開発

3つの方式は「競合」ではなく「レイヤー」が異なる。

| 方式 | レイヤー | Token効率 | 推奨 | 備考 |
|------|--------|-----------|------|------|
| Spec Kit CLI | ワークフロー管理（仕様→計画→タスク→実装） | 高 | ✅ 基本 | CLIツールのためベースラインToken増加なし。既存プロジェクトにも侵襲最小で導入可能 |
| Serena MCP | コード解析（コード⇔仕様の双方向変換） | 中 | ⚠️ オプション | 既存コードから仕様を自動抽出・整合性チェックを頻繁に行う場合のみ導入 |
| 仕様Markdown + Agent Skills | 軽量仕様管理 | 高 | ✅ 軽量変更向け | 仕様を先に固定したいがSpec Kitほどの構造は不要な変更に適する |

**推奨構成:**
```
基本: Spec Kit CLI（論点が複数ある変更・中〜大規模開発のワークフロー管理）
オプション: Serena MCP（コード解析・双方向変換が頻繁に必要な場合のみ）
軽量変更: 仕様Markdown + Agent Skill（仕様を先に固定したい軽量変更）
小改修: CLAUDE.md + Skills のみ（バグ修正・1〜2ファイルの修正）
```

**使い分けトリガー:**
```
バグ修正・小改修（1〜2ファイル）       → CLAUDE.md + Skills のみ
軽量変更（仕様を先に固定したい変更）   → 仕様Markdown + Skills
  例: 要件が明確な新機能、簡易な仕様変更、設定変更など
論点が複数ある変更 / 中〜大規模変更    → Spec Kit フルフロー
大規模リファクタリング                 → Spec Kit + Constitution見直し
```

> **補足:** 「仕様Markdown + Skills」は新機能に限定されない。簡易な仕様変更のように要件が明確・単純だが仕様を先に書きたい変更にも有効（適用例はセクション1.3）。

> **📝 運用メモ:**
>
> - **共有/ローカル分離パターン**: `.mcp.json`（project scope・git追跡）には UIコンポーネントMCP のみ登録。Serena は `--project` に絶対パスが必要なため、各自の local scope（`claude mcp add --scope local`）に設定する運用に落ち着いた
> - **用途**: コード構造の探索・シンボル検索に有効（既存コードベースの把握が速い）。一方、インフラ操作は AWS CLI が適切
> - **設定例（ローカル専用）**:
>   ```json
>   {
>     "mcpServers": {
>       "serena": {
>         "command": "uvx",
>         "args": ["--from", "git+https://github.com/oraios/serena",
>                  "serena", "start-mcp-server",
>                  "--context", "ide",
>                  "--project", "/path/to/your/project"]
>       }
>     }
>   }
>   ```

### 2.3 Git / GitHub

| 方式 | Token効率 | 推奨 | 備考 |
|------|-----------|------|------|
| `git` CLI | 高 | ✅ | Claude Code が標準で `git` コマンドを実行可能 |
| `gh` CLI | 高 | ✅ | PR作成、Issue操作等のGitHub操作 |
| GitHub MCP | 低 | ❌ | `git` / `gh` CLI で完全に代替可能。依存を増やす利得がない |

**推奨構成:**
```
git CLI + gh CLI のみ。MCP不要。
必要に応じて Agent Skill でコミットメッセージ規約等を教える。
```

### 2.4 ファイル操作

| 方式 | Token効率 | 推奨 | 備考 |
|------|-----------|------|------|
| Claude Code 内蔵ツール（Read/Write/Edit/Glob/Grep） | 最高 | ✅ | 専用ツールとして最適化済み |
| Shell (ls, rg 等) | 高 | ✅ | 内蔵ツールで不足する場合の補完 |
| Filesystem MCP | 低 | ❌ | Claude Code の内蔵ツールで完全に代替可能。依存を増やす利得がない |

### 2.5 データベース

| 方式 | Token効率 | 推奨 | 備考 |
|------|-----------|------|------|
| DB CLI (`psql`, `mysql`, `sqlite3`) | 高 | ✅ | 単発クエリ、マイグレーション |
| DB MCP | 中 | ⚠️ | スキーマ探索で有利な場合あり。ただし大きなスキーマを繰り返し取得するとレスポンスが膨らむ（§1.1 の軸②） |

**推奨構成:**
```
基本: DB CLI + Agent Skill（クエリパターンやスキーマ情報をSkillに記載）
複雑な探索: DB MCPを検討（スキーマの構造化取得が頻繁な場合）
```

### 2.6 Docker / インフラ

| 方式 | Token効率 | 推奨 | 備考 |
|------|-----------|------|------|
| `docker` / `docker compose` CLI | 高 | ✅ | 完全にCLIで運用可能 |
| Docker MCP | 低 | ❌ | `docker` CLI で完全に代替可能。MCP不要 |

## 3. 全体推奨構成（Token効率最優先）

```
Claude Code プロジェクト
├── MCP（必要最小限のみ）
│   └── Serena MCP        ← コード⇔仕様の双方向変換が必要な場合のみ（オプション）
│
├── CLI（メインのツール層）
│   ├── git / gh           ← VCS操作
│   ├── playwright         ← E2Eテスト
│   ├── docker compose     ← コンテナ管理
│   ├── psql / sqlite3     ← DB操作
│   └── npm / pip 等       ← パッケージ管理
│
├── Claude Code 内蔵ツール（最もToken効率が良い）
│   ├── Read / Write / Edit / Glob / Grep  ← ファイル操作
│   └── Bash                                ← CLI実行基盤
│
└── Agent Skills（CLIの使い方をAIに教える）
    ├── playwright-testing/SKILL.md
    ├── db-operations/SKILL.md
    └── deploy-workflow/SKILL.md
```

### 原則まとめ

| # | 原則 | 理由 |
|---|------|------|
| 1 | **Claude Code 内蔵ツールを最優先** | 専用最適化されており、返る情報量も制御しやすい |
| 2 | **次にCLI + Agent Skills** | 必要時のみSkillが読まれる。ベースライン増加なし。stdout で必要な分だけ返せる |
| 3 | **MCPはCLI代替がない場合のみ** | 代替できるなら、常駐プロセス・環境依存設定・承認フローという依存を増やす利得がない |
| 4 | **高頻度・大レスポンスの操作はCLI側へ寄せる** | tool search でベースラインは解消したが、**呼び出しごとのレスポンス量は残る**（§1.1 引用: Playwright MCP は1操作 5,000〜10,000 トークン） |
| 5 | **MCP導入数は必要な分に絞る** | ツール名・サーバー指示の常時ロード分に加え、接続・承認・保守のコストが積み上がる |

## 4. Agent Skills × CLI 連携パターン

MCPの代わりにCLIを使う場合、AIがCLIの使い方を知っている必要がある。
Agent Skillsで補うことで、Token効率を維持しつつ、MCPと同等の使い勝手を実現できる。

### パターン例：Playwright CLI用 Skill

```markdown
---
name: playwright-testing
description: Playwright CLIでE2Eテストを実行・管理する。テスト実行、スクリーンショット取得、テスト作成時に使用。
---

# Playwright テスト運用

## 基本コマンド

テスト実行: `npx playwright test`
特定ファイル: `npx playwright test tests/login.spec.ts`
UI モード: `npx playwright test --ui`
レポート: `npx playwright show-report`

## スクリーンショット

`npx playwright screenshot <url> <output.png>`

## テスト生成

`npx playwright codegen <url>`
```

**この構成のToken効率:**
- MCP版: ツール定義が常時コンテキストに存在（数百〜数千Token）
- Skill + CLI版: Skillはplaywrightに関する質問が来た時だけ読み込まれる

## 5. MCP導入時の設定・運用ノウハウ

MCPを導入する場合でも、Token消費を最小化する運用が重要。

### 5.1 設定ファイルの管理

**共有/ローカル分離パターン（実証済み）:**

プロジェクト共有と個人ローカルを分けて管理する。`--project` に絶対パスが必要な MCP（Serena等）は共有設定に含められないため、local scope に分離する。

`[Fact]` MCPサーバーの設定scopeは3つあり、保存先が異なる。出典: [Claude Code Docs / MCP — MCP installation scopes](https://code.claude.com/docs/en/mcp)

| scope | 読み込まれる範囲 | チーム共有 | 保存先 |
|-------|-----------------|-----------|--------|
| **local**（既定） | そのプロジェクトのみ | ❌ | `~/.claude.json`（プロジェクトパスごとに記録） |
| **project** | そのプロジェクトのみ | ✅ git追跡 | プロジェクト直下の `.mcp.json` |
| **user** | 全プロジェクト | ❌ | `~/.claude.json` |

> **⚠️ よくある混同:** MCPの **local scope** の保存先は `~/.claude.json` であり、**`.claude/settings.local.json` ではない**。後者は permissions 等の一般のローカル設定用で、MCPサーバー定義の置き場ではない。名前が似ているため取り違えやすく、公式ドキュメントもこの点を名指しで注意している。

```jsonc
// .mcp.json（project scope・git追跡 — 絶対パスを含まないMCPのみ登録）
{
  "mcpServers": {
    "ui-component-mcp": {
      // UIコンポーネント仕様参照など、パスに依存しないMCPを登録
    }
  }
}
```

```bash
# local scope へ追加（保存先は ~/.claude.json。git追跡されない）
# Serena は --project に絶対パスが必要なため共有不可
claude mcp add serena --scope local -- \
  uvx --from git+https://github.com/oraios/serena \
  serena start-mcp-server --context ide --project /path/to/your/project
```

> **`--` の意味:** stdio サーバーでは `--`（ダブルダッシュ）が Claude 自身のオプション（`--scope` 等）と、サーバーを起動するコマンド・引数を区切る。`--` 以降はそのままサーバーへ渡される。

> **注意:** `@anthropic/serena-mcp`（旧 `npx` 方式）は現在 `uvx` ベースの `oraios/serena` に置き換わっている。旧設定例は使用しないこと。

### 5.2 運用上の注意点

| 注意点 | 対策 |
|--------|------|
| **MCPのレスポンスが大きくコンテキストを圧迫する** | `[Fact]` Claude Code は出力が10,000トークン超で警告し、既定で25,000トークンに制限する（`MAX_MCP_OUTPUT_TOKENS` で変更可）。**制限に当たるようなら、その操作はCLI側へ寄せる合図**（§1.1 の軸②） |
| MCP数が増えると管理・接続コストが積み上がる | 本当に必要なMCPだけ登録し、不要になったものは即座に削除 |
| tool search が無効な環境では全ツール定義が先読みされる | 配布先の環境・モデルを確認する。無効化される条件は §1.1 参照 |
| MCPの起動に時間がかかる場合がある | 開発セッション開始時にまとめて起動確認 |
| project scope の MCP は承認が要る | `.mcp.json` のサーバーは初回に承認プロンプトが出る。チーム配布時は手順に含めておく |

## 6. ドキュメント記述規約（Token効率の実践）

ナレッジ・仕様・ルール等の作業ドキュメントを書く際の、Token効率と可読性を両立する指針。Skill/プロンプトの*出力*規約とは別軸で、ドキュメントの*記述*に関するもの。

| 原則 | 内容 |
|------|------|
| 地の文は日本語 | 論旨・判断・説明は日本語。ユーザー自身が読む・編集する正本のため可読性を優先 |
| 用語・ラベルは英語（定着語のみ） | `Token` / `API` / `Output Format` / `[Fact]` 等の定着した用語は英語。馴染みの薄い一般語をToken目的で英語化しない（翻訳コストで逆効果） |
| 削減は「英語化」でなく「冗長削減」 | 日本語はToken密度が高いが、削減の主レバーは言語選択より簡潔さ・構造。まず冗長な記述を削る |
| 外部リソースは human-readable 名を主に | チケット・ページ等を数値ID単独で書かない。名前を主に、IDは補助、理想は markdown リンク `[名前](URL)` |
| 読者可読性を優先し内輪語を避ける | 専門外の読者に伝わらない略語・外来語を避け、平易な日本語で書く。略語は初出で正式名称を書く（例: `AC` →「受入基準（Acceptance Criteria）」）。定着した英語技術用語（`commit` / `push` / `API` 等）は対象外。一覧は `01`「略語一覧」に集約。特に公開・共有ドキュメントでは、確定前に可読性を1パス点検する |

> **公開・共有前の1パス点検:** 確定・公開の前に ①可読性（略語の正式名称化・不要な外来語）②機微情報（固有名・個人情報・特定できる第三者への評価）③記述と実体の一致（説く運用をリポジトリ自身が満たしているか）をまとめて点検する。

> **実証（カタカナ語英語化の見送り）:** 「英語をカタカナにしただけ」の定着語（ターゲット→target 等）を英語へ戻す案は、①得が小さい ②英語綴りは文字数が増え可読性がむしろ下がる ③冗長削減の方が効く、との理由で見送った。削減は英語化ではなく冗長削減に集約するのが原則。
