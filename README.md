# AI駆動開発 実践ガイド

**アジャイル/スクラムの規律を、AIとの協働に適用する。**

Claude Code / Agent Skills を活用したAI駆動開発（AIDD: AI-Driven Development）の**方法論と実践知を体系化した**実践ガイド。ツールの使い方集ではなく、**進め方（プロセス）の規律**を主題とする。

## 短時間で中身を確かめたい方へ

全部を読む前に、このガイドが何を大事にしているかを判断できる3点。

1. **[docs/design-decisions.md](docs/design-decisions.md)** — なぜこの構造なのかの記録。**自分の誤りを訂正した記録を含めてある**（裏取り済みの記述でも読み違えれば誤る、引用を自己の実証として提示していた、など）。方法論の文書が生成物と区別されるのは、こうした記録が残っているかどうかによる
2. **[BACKLOG.md](BACKLOG.md)** — **まだ検証できていないことの一覧。** どの章がどこまで裏取り済みかを表にしてある。実証済みと未検証の混在を隠さないことが、本ガイド自身への確度ラベルの適用にあたる
3. **[01](01_ai-driven-dev-strategy.md) セクション2** — 確度ラベルとクロスレビュー。**「もっともらしい散文」を反証可能な状態に落とす**ための中核

## 背景・設計思想

このガイドは、AIツールの使い方集ではなく、**アジャイル/スクラムの実践知を土台に、AIネイティブ開発を規律づける**試みである。ここでの規律の柱は、仕様駆動開発（SDD: Spec-Driven Development、何を作るかを先に仕様として固める進め方）とテスト駆動開発（TDD: Test-Driven Development、先にテストを書いてから実装する進め方）の2つである。

著者は、経験豊富なアジャイルコーチや、「真のアジャイル」を志向して集まった認定スクラムマスター／プロダクトオーナーとのチームでの実践を通じて、メタ認知とクリティカルシンキングを鍛えられた。良いスクラムも不完全なスクラムも経験し、フルスタックエンジニアとしてチームをボトムアップで正しいスクラムへ導く役割（スクラムマスター兼開発者）も担ってきた。

この土台が、本ガイドの一貫した思想になっている。

- **人間が裁定者**：AIの提案を鵜呑みにしない。スクラムの「検査と適応」を、人間の判断責任として貫く（[01](01_ai-driven-dev-strategy.md) セクション2）
- **曖昧さを検査可能にする**：確度ラベル（`[Fact]` / `[Judgment]` 等）と受入基準（AC: Acceptance Criteria）で、「もっともらしい散文」を反証可能な状態へ落とす（[01](01_ai-driven-dev-strategy.md) §2 ／ [06](06_requirements-spec-brainstorm.md) §5）
- **WHAT と HOW の分離**：作るもの（What）と作り方（How）を分ける。テクニカル PdM（プロダクトマネージャー。技術的判断まで踏み込む立場）として、要件・仕様（何を作るか）の壁打ちまでを射程に収めた WHAT レイヤー（[06](06_requirements-spec-brainstorm.md)）を持つ

そのため本ガイドは、フルスタック向けの AIネイティブ／SDD／TDD の HOW（`01`〜`05`）に加え、PdM 視点の WHAT（`06`）を別レイヤーとして備える。WHAT レイヤー（`06`）は、要件・仕様が曖昧なまま開発が進み全体像が見えなくなる実プロジェクトの経験から生まれた——「何を作るか」を検査可能な形で固める必要を痛感したことが、この章の起点である。

## 壁打ちの進め方

### Token効率を最大化する運用ルール

1. **トピック別にセッションを分ける** — 1つの長いセッションを続けない。議論が一区切りついたらセッションを閉じる
2. **確定した内容はMarkdownに書き戻す** — Markdownが「外部記憶」として機能し、次のセッションで同じ議論を繰り返さない
3. **`@ファイル名` で必要なファイルだけ参照** — 全ファイルを同時に渡さない
4. **壁打ち済みの箇所は確定マーク** — `<!-- BRAINSTORM -->` を消して確定内容に置き換える

### セッションの始め方・終わり方

セッションは**スプリント**として扱う（[01](01_ai-driven-dev-strategy.md) §3.6）。前セッションの終わりにレトロスペクティブを行い、次セッションの冒頭で前回 Action を検証（**Check Action**）してから作業に入る。この往復がないと、決めた改善が願望のまま蒸発する。

**始め方**（再開手順の実体は [CLAUDE.md](CLAUDE.md)「再開手順」）

1. `CLAUDE.md`（現在モード）と**自分の作業日誌**（次にやること・現在地）を確認する
2. **Check Action** — 前回の Action がやれたか／結果はどうかを検証する。併せて、前回 `[Assumption]`（未確認の推測）で書いた箇所を再点検する
3. 着手タスクが名指すファイルだけを `@` 参照する（`01`〜`06` を一括ロードしない）

**終わり方**

4. 確定内容を該当ファイルへ、判断を [docs/design-decisions.md](docs/design-decisions.md)、残課題を [BACKLOG.md](BACKLOG.md)、作業の現在地を作業日誌へ書き戻す
5. 節目でレトロスペクティブを行い、次セッションの Action を作業日誌に置く

> **作業日誌は配布されません。** 「次にやること」「作業の現在地」「稼働の記録」は**書き手の生活の記録**であって方法論ではないため、公開対象から外しています（理由は [docs/design-decisions.md](docs/design-decisions.md)「作業日誌は公開しない」）。**このガイドを使う人は、自分の作業日誌を自分で持ってください**——`clone` した直後なら [BACKLOG.md](BACKLOG.md) を起点にし、最初のセッションで作るのがおすすめです。

### モデル選択の指針

| 目的 | 推奨モデル | 具体的ユースケース |
|------|-----------|------------------|
| 深い概念整理・設計の壁打ち | Claude Opus 4.8 | 大規模リファクタリングの設計、ドキュメント体系の壁打ち |
| テンポの良い反復議論 | Claude Sonnet 5 | 方針確定後の詳細化、.mdファイルの更新作業 |
| クロスレビュー・再検証 | まず新セッション＋`@`参照（同一モデルで可）／できれば別系統の高性能モデル | 前提を追認させない再検証、仕様の整合性チェック |
| 大量コンテキストの俯瞰 | 大容量コンテキスト特化モデル | 大規模コードベースの俯瞰、長大コンテキストの一括把握 |

> **既定モデルと切替の原則:** 判断主体の作業（設計・壁打ち・レビュー）は高性能モデルを既定にし、大量の機械的編集フェーズのときだけ高速モデルへ手動で切り替える。詳細は [01_ai-driven-dev-strategy.md](01_ai-driven-dev-strategy.md) セクション2「既定モデルと切り替えの原則」。

> **クロスレビューに複数ベンダーのアカウントは要らない:** 本質は「別モデル」ではなく、**推論過程を見ていない独立した目**で読み直すこと。①新セッション＋`@`参照 ②レビュー用 Skill ③別系統の高性能モデル の3段階で確保でき、**段階1だけでも成立する**（段階3は「できれば」）。詳細は [01_ai-driven-dev-strategy.md](01_ai-driven-dev-strategy.md) セクション2「クロスレビュー運用」。

## 別マシンでの再セットアップ

このリポジトリが Claude Code 運用設定の **正本** であり、プロジェクト共通設定はリポジトリ内（`CLAUDE.md`）、個人用設定は `~/.claude` に置く。新しいマシンで同じ運用に戻すには以下の手順で十分。

> **正本は git 追跡ファイル、AIのメモリは per-machine キャッシュ:** 耐久的な知識（規約・決定・プロジェクト文脈）は必ず git 追跡ファイル（本リポジトリの `.md`）に置く。Claude Code の自動メモリ機能はマシン単位のキャッシュで `git clone` に付いてこないため、メモリだけに残すと別マシンで文脈が途切れる。メモリは併用してよい（重複許容）が、正本は常に git 側に置く。

**手順の正本は [setup-claude-code.md](setup-claude-code.md)。** 環境の定義（どこに何を置くか）と再現手順はすべてそちらにある。**ここには手順を列挙しない**——README と手順書に同じ列挙を置いていた時期があり、実際に README 側が腐った（作業日誌への言及が抜け落ちた）。

> **⚠️ clone するのは1つではない。** このリポジトリだけを clone しても、`CLAUDE.md` の再開手順は空振りする（作業日誌が無いため）。**自分の作業日誌リポジトリを並置する**必要がある。詳細は [setup-claude-code.md](setup-claude-code.md) の手順 1-2。

## 生成AIの利用について

本ガイドは、生成AIとの壁打ち・クロスレビューを通じて作成した。具体的なプロセスは以下の通り。

- **壁打ち・初稿:** Claude Opus 4.8 による議論・構造化
- **クロスレビュー:** 別系統の高性能モデルによる再検証・セカンドオピニオン
- **精査・裁定:** 著者（人間）が全ての方針決定・取捨選択を実施
- **検証・改善:** 実際の開発への適用を通じて継続中（どこまで裏取りできているかは [BACKLOG.md](BACKLOG.md) の「章ごとの裏取り状況」を参照）

AIの出力をそのまま採用したものではなく、複数モデルの視点を突き合わせたうえで人間が判断している。ただし、内容の正確性を完全に保証するものではない。

## 著者について

アジャイル/スクラムの実践を土台に、AIとの協働を規律づけることに関心があるフルスタックエンジニア。

- **スクラムマスター** — **その役割で参画した経験**と、**開発者としてチームをボトムアップで正しいスクラムへ導いた経験**の両方がある。良いスクラムも不完全なスクラムも経験している
- **テクニカル PdM** — 技術的判断まで踏み込む立場で、要件・仕様（何を作るか）を検査可能な形に固める仕事をしてきた。本ガイドの WHAT レイヤー（[06](06_requirements-spec-brainstorm.md)）はここから生まれた
- **経験豊富なアジャイルコーチ、認定スクラムマスター／プロダクトオーナーとのチーム**での実践を通じて、メタ認知とクリティカルシンキングを鍛えられた

本ガイドが「ツールの使い方集」ではなく「進め方の規律」を主題にしているのは、この経歴の反映である。
