# Claude Code セットアップガイド

新しいマシンで同じ Claude Code 運用を再現するための手順書。

## 正本と展開先の関係

**環境の定義は、この表を単一の正本とする。** 他のファイルに手順を重複させない（二重管理は必ず片方が腐る）。

| 場所 | 役割 |
|------|------|
| **このリポジトリ** | 正本。ナレッジ・ルール・テンプレートの管理元 |
| **作業日誌リポジトリ**（並置・非公開・**各自が持つ**） | 「次にやること」「現在地」「Check Action の結果」を置く。**このリポジトリには含まれず、clone しても付いてこない**（理由は `docs/design-decisions.md`「作業日誌は公開しない」） |
| **`~/.claude`** | 展開先。個人メモリ・個人用Skill・ログ等を置く。Git 管理しない |

> **AIの自動メモリは正本にしない:** Claude Code の自動メモリ機能はマシンの絶対パス依存の per-machine キャッシュで、`git clone` に付いてこない。耐久的な規約・決定・プロジェクト文脈は必ずこのリポジトリの git 追跡ファイル（`CLAUDE.md` や `.md` 群）に書き、メモリはキャッシュとして併用する（メモリが無くても文脈が途切れない状態を保つ）。

## セットアップ手順

### 1. リポジトリを clone する

```bash
git clone <repository-url> ~/git/aidd-practice-guide
```

clone すると、リポジトリルートの `CLAUDE.md`（プロジェクトメモリ・常時ロード）がそのまま有効になる。

### 1-2. 作業日誌リポジトリを並置する

> **⚠️ ここを飛ばすと再開手順が空振りする。** `CLAUDE.md` の再開手順は「作業日誌『次セッションの開始点』を読む」から始まるが、**作業日誌はこのリポジトリに含まれない**（理由は `docs/design-decisions.md`「作業日誌は公開しない」）。

**「並置」とは、本体リポジトリと同じ親ディレクトリに、兄弟として置くこと。**

```
~/git/
  ├── aidd-practice-guide/   ← 本体（このリポジトリ）
  └── aidd-worklog/          ← 作業日誌（非公開・各自が持つ）
```

- **まだ持っていない場合:** 上記の位置にディレクトリを作り、`git init` して `worklog.md` を置く。中身は `BACKLOG.md` を起点に、最初のセッションで作ればよい。git 管理下に置くのは、**AI に書かせた内容を差分でレビューするため**（`docs/design-decisions.md`「非公開ファイルは『git 管理外』ではなく『別リポジトリ』に置く」）
- **既に持っている場合（別マシンからの移行など）:** 上記の位置に clone する。**private リポジトリなら、clone の前にその環境で認証を通しておく**（`gh auth login` 等）。公開リポジトリと違い、認証なしでは clone 自体が失敗する

> **📌 ディレクトリ名は固定値である。** `CLAUDE.md` は作業日誌の実体を **`../aidd-worklog/worklog.md`** という相対パスで持っている。**動作に効くのはその1行だけ**なので、別の名前・別の場所に置くなら、そこを自分の環境に合わせて書き換える（**この手順書に出てくる同じ名前は説明であって、動作には影響しない**）。

### 2. ナレッジを読む

`README.md` を起点に、ドキュメント構成（`01` 〜 `06`）を番号順に確認する。
全文を読む必要はなく、概要と自分の関心のあるセクションだけで十分。

### 3. ユーザーメモリを設定する

[user-memory-template.md](user-memory-template.md) の本文を `~/.claude/CLAUDE.md`（ユーザーメモリ）に反映する。
これは全プロジェクト共通の個人作業原則で、Claude Code が常時ロードする。

要点:
- `~/.claude/CLAUDE.md` が無ければ新規作成し、テンプレートの本文を追記する
- 「日本語で応答する」原則が未設定なら追加し、既にあればそのまま残す

### 4. 個人用 Skill を配置する（任意）

`~/.claude/skills/<skill-name>/SKILL.md` に個人用 Skill を配置する。
そのリポジトリ（または手元のバックアップ）から再配置する。現時点では手動コピーで十分。

プロジェクト固有の Skill は、このリポジトリの `.claude/skills/` に置けば clone だけで共有される。

### 5. プロジェクト固有の設定（任意）

タスク固有の短命コンテキストファイル（例 `docs/tasks/*.md`）は短命ファイルとして扱い、初回セットアップ対象には含めない（タスク開始時に作成し、PR作成前に削除する）。

### 6. セッションを開始する

本体リポジトリのルートで Claude Code を起動し、再開の合図を送る（著者は `go` の1語で運用している）。`CLAUDE.md`「再開手順」が発火し、**現在モードの確認 → 作業日誌『次セッションの開始点』の読み込み → Check Action** の順に進む。

**ここまで到達できれば、セットアップは完了している。** 逆に、作業日誌が並置されていないと手順の2つ目で止まる——**それがこのガイドの実質的な動作確認になる**。

## 同期しないもの

以下は `~/.claude` 内で自動生成・マシン固有のため、マシン間で同期する必要はない。

- 会話ログ・トランスクリプト等の自動生成ファイル
- 絶対パスを含むローカル専用設定（`.claude/settings.local.json` の permissions 等）
- MCPサーバーの local / user scope 定義（`~/.claude.json`）。絶対パスを含むものはマシンごとに `claude mcp add --scope local` で入れ直す（`02` §5.1）

## 将来の自動化について

現時点では自動化スクリプト（`setup.sh` 等）は **作成しない**。理由:

- セットアップ手順が少なく、手動で十分完了できる
- 絶対パスやローカル依存の設定がまだ少ない

以下の条件が揃った場合に自動化を検討する:

- 複数の個人用 Skill を `~/.claude/skills/` に配布する運用が定着した時
- 毎回同じ手作業が **3回以上** 繰り返された時
- 環境固有の差分（パス・認証情報等）が増えて手動設定の負担が無視できなくなった時
