# 設計判断の記録（design decisions）

本ガイドが「なぜこの構造・この規律なのか」を残す記録。手順そのものは `01`〜`06` にあり、ここには**その手順を選んだ理由**と、**選ばなかった選択肢**を置く。

> **この記録の役割:** 方法論は、結論だけを読むと単なる作法に見える。理由が残っていれば、前提が変わったときに読み手が自分で判断し直せる。本ガイド自身が説く「What と根拠を検査可能にする」規律の自己適用である。

---

## 1. 構造の設計判断

### 方法論の背骨をアジャイル/スクラムの実践知に置く

- **決定:** 本ガイドの土台を「アジャイル/スクラムの実践知＋クリティカルシンキング」に置き、それを AI との協働に適用する。具体的には ①人間を裁定者とする ②確度ラベルと受入基準（AC）で曖昧さを検査可能にする ③WHAT と HOW を分離し、WHAT（要件・仕様）を別レイヤーとして持つ（`06`）。
- **理由:** AI を使う開発でも、必要なのは「検査と適応」と「人間の判断責任」という規律である。**ツールは変わるが、この土台は変わらない**——本ガイドが特定ツールの Tips 集ではなく進め方の規律を主題に据えているのはこのためである。

### 番号順 = 読み順（01〜06）

- **決定:** ファイルを推奨読み順に採番する。`01 → 02 → 03 → 04 → 05` が開発の HOW、`06` は WHAT の別レイヤー拡張。
- **理由:** 読み手が「どこから読むか」を迷わないため。HOW（どう作るか）と WHAT（何を作るか）は思考のレイヤーが異なるため、`06` は連番の続きではなく別レイヤーとして分離した。

### 方法論（methodology）とプロジェクト固有情報（content）を分離する

- **決定:** 再利用可能な方法論（手順・観点・行動規範）と、プロジェクト固有の情報（要件・数値・対象名）を役割で分ける。方法論は Skill や本文に、固有情報は仕様ファイルや `CLAUDE.md` に置く。
- **理由:** 両者を混ぜると、Skill やナレッジが特定プロジェクトに癒着して横断再利用できなくなる。可変値を固有情報側に隔離することで、方法論を恒久的な資産として保てる。

### 単一の「再開用メモ」を廃止し、記憶モデルを役割で分ける

- **決定:** 1つのファイルに「現在状態・ロードマップ・決定ログ・課題索引・再開手順」を詰め込む運用をやめ、役割ごとに分離した。制御plane → `CLAUDE.md` ／ 決定の記録 → 本ファイル ／ 発展課題 → `BACKLOG.md`。
- **理由:** 単一メモは、**古びる部分（状態）と伸び続ける部分（ログ）が同居する**ため形骸化し、更新の起点が定まらなくなる。役割を分ければ、常時ロードされる `CLAUDE.md`・本ファイル・`BACKLOG.md` の3点だけで文脈が復元でき、単一の再開メモは不要になる。
- **git 履歴を文脈復元の手段に数えない:** コミットメッセージの粒度は書く時々の余力に左右され、**均一を期待できない**（本ガイド自身の履歴でも、詳細な本文を持つコミットと要約1行だけのコミットが混在した）。加えて、書き戻しの規律を守っていれば**コミット本文の内容は `.md` 側に重複して存在する**——git 履歴は一次情報ではなく複製である。文脈の復元は、粒度を自分で保証できる `.md` に依存させる。

### 作業日誌は公開しない（稼働の記録と、内容についての判断を分ける）

- **決定:** 「次にやること」「作業の現在地」「改善 Action の検証結果」「稼働の記録」を**作業日誌**として分離し、公開対象から外す。公開するのは**内容についての判断**（本ファイル）と**未解決の課題**（`BACKLOG.md`）に限る。
- ⚠️ **置き場所はこの後に変更した。** 当初は `docs/worklog.md` を `.gitignore` で除外する形で実装したが、**除外では変更を差分で確認できない**という問題が出たため、別リポジトリへ移した（次項）。**分離するという決定自体は変わっていない**——変わったのは実装方法だけである。
- **判定基準:** 「内部か外部か」ではなく **「著者の生活・稼働の記録か、内容についての判断の記録か」**。
- **理由:** 作業日誌には、個々には無害でも**時系列で累積すると別の情報になる**記述が溜まる——いつ作業したか、どれだけ続けたか、どこで止まったか。これは方法論としての価値を持たない一方、書き手の生活を再構成できてしまう。**エントリ単位で「これは公開してよいか」を判定する方式では、この累積を検出できない。** 分離はファイル単位で行う必要がある。
- **公開版に残した理由（`BACKLOG.md`）:** 未解決論点と「まだ発火していない方法論」の一覧は、**内容の確からしさの境界を示すもの**であり、読者にとって有用である。実証済みと未検証の混在を隠さないことは、確度ラベルの規律の延長にある。
- **読者にとっての意味:** 本リポジトリを clone しても作業日誌は付いてこない。**作業日誌は各自が自分で持つもの**であり、配布する種類の資産ではない——これは制約ではなく設計である。

### 非公開ファイルは「git 管理外」ではなく「別リポジトリ」に置く

- **問題:** 上記の分離を `.gitignore` で実装した結果、**作業日誌と決定ログが git 管理から外れ、セッションごとの変更を差分で確認できなくなった**。AI に書き戻しをさせている以上、これは実害がある——**レビューできない書き戻しは、正本の品質を検査の外に置く**ことになる。
- **決定:** 非公開ファイルは **独立したリポジトリ**（並置したディレクトリ）に置き、そこで git 管理する。公開リポジトリからは**一切参照しない**。
- **git サブモジュールを採らなかった理由:**
  1. **機能が用途に合わない。** サブモジュールの本質は「親の各コミットが子の特定コミットをピン留めする」ことであり、**バージョンの対応付けが要る場合に見合う複雑さ**である。作業日誌にその要求はない（「本文が過去のある版だったときの作業日誌」を復元したい場面が存在しない）。**使わない機能のために複雑さだけを払う構成になる。**
  2. **`.gitmodules` は公開される。** 非公開リポジトリを指すポインタが配布物に残り、clone した読者の取得が失敗する。「**clone だけで運用を再現できる**」という本ガイドの原則と正面から衝突する。
  3. **「AI が適用忘れを防ぐ」は保護として弱い。** 本リポジトリでは、AI に特定ファイルを読ませない設定が別経路で破れた実測がある（下記「守るコストが高い規律は形骸化する」）。**ツールの振る舞いに依存する保護は、その前提が崩れた瞬間に無防備になる。**
- **一般化できる原則:** **「守るコストが高い規律は形骸化する」の系として、設定や規律で塞ぐより、構造で成立させるほうが強い。** 本ガイドではこの原則が2度適用された——①AI に読ませたくない下書きは、読取拒否の設定ではなくリポジトリ外への配置で解決する ②公開したくないファイルは、`.gitignore` による除外ではなく別リポジトリでの管理で解決する。**いずれも「禁止する」から「そもそも同じ場所に置かない」への移行**である。
- **実装後の観測 `[Fact]`（実施セッションで実測）:**
  - **作業ディレクトリの外にあるファイルも、AI から読み書きできた。** 「リポジトリの外に出すと AI が扱えなくなるのでは」という懸念は成立しなかった。分離のコストは想定より小さい。
  - **コミットが2回になる**（公開側と作業日誌側）。これは副作用ではなく**狙った効果そのもの**である——書き戻しが差分として現れることが分離の目的だった。
  - **制御plane側の実装で効いた工夫:** 参照は「**作業日誌**」という**役割名**で持ち、実体パスは**1箇所にだけ**書く。こうすると、読者にとっては「各自が自分の場所に持つもの」として読め、書き手にとっては移動時の修正が1箇所で済む。**パスを本文に散らすと、非公開ファイルの置き場所を変えるたびに公開物が壊れる。**

### 稼働の記録は落とすが、原理としての言及は残す（疲労への言及の扱い）

- **決定:** 「疲労は人間にしか読めない信号である」という**原理としての言及**（`01` §3.6 の分担表）と、その原理を導いた**単発の出来事の引用**（「疲れたのでミスが頻発しそうだと判断して終了を促した」）は、**いずれも公開版に残す**。
- **判定:** 上記「作業日誌は公開しない」の基準——**「書き手の生活・稼働の記録か、内容についての判断の記録か」**——を適用した。両者は**いつ・どれだけ働いたかを示さない**。示しているのは「**なぜ終了判断を人間側に残す設計にしたか**」という根拠であり、内容についての判断にあたる。
- **落としたもの:** 同じ出来事でも、日時・連続時間・頻度・体調の推移といった**再構成可能な形の記述は作業日誌に置き、公開しない**。分離の境界は題材ではなく**粒度**にある。
- **なぜこの線引きが要るのか:** 疲労への言及を一律に禁じると、**分担設計の根拠が消えて「人間が裁定する」が理由のない規則になる**。方法論の文書では、**規則よりも規則を成立させた観測のほうが再利用価値が高い**——根拠を落とすと、読者は自分の状況に合わせて規則を調整できなくなる。

### 正本は git 追跡ファイル、AI のメモリはキャッシュ

- **決定:** 耐久的な知識（規約・決定・プロジェクト文脈）は必ず git 追跡ファイルに置く。AI ツールの自動メモリ機能は併用してよいが、正本にはしない。
- **理由:** 自動メモリはマシン単位のキャッシュであり `git clone` に付いてこない。メモリだけに残すと、別マシンで作業を再開した瞬間に文脈が途切れる。

## 2. 規律の設計判断

### 確度ラベルを重要度とは別軸で導入する

- **決定:** 壁打ち・レビューで、重要度（🔴🟡⚪）とは別に確度ラベル（`[Fact]` / `[Judgment]` / `[Assumption]` / `[To Be Verified]`）を併用する。
- **理由:** 「どれだけ効くか（重要度）」と「どれだけ確かか（確度）」は直交する軸である。分けて示すことで、断定してよい指摘と要確認の指摘が混ざらず、AI の創作的な断定を検査にかけられる。

### 人間が採否を判断しうる出力には、すべて一意な ID を振る

- **決定:** レビュー・監査の指摘に一意な ID（`F1`, `F2`…）と状態列（採用／却下／保留／次へ送り）を与え、表形式で提示する。適用範囲は指摘だけでなく、**提案・意見・前言撤回・選択肢・要確認事項を含む「人間が採否を判断しうる出力単位すべて」**とする。
- **理由:** 見出しと重要度だけの散文は、人間が引用して「これは採用、これは却下」と指示しづらい。安定した ID があれば項目単位の裁定と追跡が軽くなり、人間が裁定者として回す規律を実務的に支える。
- **範囲を広げた経緯:** 当初は「レビュー・監査の指摘」に限定していたが、その結果 AI の提案・意見・撤回が規律の外に落ちて散文へ逆戻りする事象が繰り返された。**規律は狭く定義すると抜け道ができて機能しない。**

### クロスレビューの本質は「別ベンダー」ではなく「独立性」

- **決定:** 初稿を書いた主体を唯一の裁定者にしない。ただし要件は**独立性の確保**（推論過程を見ていない目で読み直すこと）であり、別ベンダーのモデルは前提条件ではなく最上位の手段と位置づける。①新セッション＋`@`参照 ②レビュー用 Skill ③別系統モデル の3段階に構造化した。
- **理由:** 単一モデルは自己の出力を過大評価する。一方で、複数ベンダーのアカウントを持たない読者でも段階1で成立し、**追認バイアスの除去という効果の大半は追加コストなしで得られる**。

### レトロの実効性を、人間の記憶ではなく手順で担保する

- **決定:** 前回の改善 Action の検証（Check Action）を `CLAUDE.md` の再開手順に組み込み、セッション冒頭で機械的に実行する。あわせて Try 表に「処理」列、Action 表に「由来」列を必須化し、全 Try の行き先と Action の出所を追えるようにする。
- **理由:** 前回 Action の検証を人間の意思に委ねた結果、実際に飛ばされ、決めたはずの Action が守られないまま作業が進んだ。**規律は人間依存ではなくファイルの縛りに移す。** また ID を振っても Try と Action の対応を示さなければ、読み手が突き合わせを再構成する負荷を負う——**ID を振るだけでは不十分で、ID 同士を繋ぐことまでが要件**である。

### 人間側の防御は「注意力」ではなく「時間差」で設計する

- **決定:** Check Action に「前セッションで正本に書いた主張を再点検する」手順を含める。対象は未確認の推測に基づくものだけでなく、**裏取り済みの記述も含む**。
- **理由:** 「鵜呑みにしそうだと感じたら保留する」という心がけは、**疲れているときに最初に壊れる**ため単独では機能しない。実際、誤った記述がその場のレビューを通過し、後から背景情報が加わって露見した事例がある。意志ではなく手順で拾う形にした。
- **この検査を AI に代替させない理由:** 検査されなかった主張の発生源に、その検査の監督を任せることになり、「初稿を書いた主体を唯一の裁定者にしない」と同じ理由で破綻する。AI にできるのは判断の肩代わりではなく、確度ラベルと ID による**検査しやすさの提供**までである。

### 改善 Action は常時2件までとし、超えたら捨てる

- **決定:** レトロから持ち出す改善 Action は2件を上限とする。3件目以降は記録は残すが追跡項目にはしない。
- **理由:** **追跡されない Action は、無い Action と同じ**である。件数を増やすより、2件を確実に検証するほうが改善は進む。この規律は導入した回のレトロで自己適用し、挙がった5件をその場で2件に絞った。
- **意図していなかった副次効果（数セッション後に観測）`[Judgment]`:** **上限がセッションの長さとスコープを制約するガードレールとしても働いた。** 追跡項目が2件しかないと、1セッションで扱える改善の幅が構造的に決まり、際限なく論点を広げられなくなる。**この効果は設計時に意図していなかったが、主効果より重要かもしれない**——「レトロだけを輸入するとプロセスは洗練されるが出荷されない」という診断に対して、**改善の項目数を制限することが、結果として出荷への集中を生んだ**。
- **枠を増やさない理由:** 「1件くらい増やしてもよいのでは」は繰り返し起こる。しかし**追跡したい改善が先にあって枠が足りないのでなければ、枠は埋まる方向にしか働かない**。上限を触るのではなく、**達成した Action を卒業させて回転させる**ことで実質的な容量を確保する。
- **常設手順は Action に数えない:** 毎回実行する手順（正本に書いた主張の再点検など）には「卒業」という概念がなく、**Action 枠に置くと永久に1枠を占有する**。これらは再開手順側に一本化し、改善 Action の枠を空ける。**同じ項目を Action 枠と手順の両方に置くのは二重管理である。**

### 改善 Action の卒業は「削除」ではなく「規約への統合」で行う

- **決定（2026-07-21）:** 改善 Action が卒業条件を満たしたら、Action 枠から**削除するのではなく `CLAUDE.md` の常設規約へ統合する**。実例＝N4（著者の環境・既往作業の**状態**は実測／本人確認の後に発話する）を、原因調査が実際に発生したセッションで ○ を取ったため卒業させ、「役割の断定に確度ラベルを付ける」規約の**姉妹規約**として統合した。
- **理由:** Action 枠（常時2件）は**追跡のための枠**であって、達成した規律の保管場所ではない。達成したら常設規約へ移して枠を空ける。**削除すると規律ごと消える**——卒業は昇格であって除外ではない。除外による除去（N2＝役割終了）とは区別する。
- **併せて（採番の規約化）:** レトロの各項目に一意な ID を付ける規約を明文化した。**それまで採番は過去レトロの前例（precedent）としてしか存在せず、規約化されていなかったため、あるセッションで黙って落ちた**（人間が指摘して発覚＝K-6 の型）。**precedent だけの規律は発火の合図を持たず、静かに腐る**——「実行対象のない規約」「発火の合図がない規約（対話での呼称）」に続く**第3の欠陥型**として記録する。

### セッション末に「成果物は前に進んだか」を1問だけ検査する

- **決定:** セッションの終わりに検査の1問を置き、No が続くならそれ自体を最上位の課題として扱う。当初の問いは「このセッションで成果物は公開／利用に近づいたか」だった。
- **理由:** レトロだけを回していると、**プロセスの改善が進捗の代わりになる**。セッション内の実行品質は検査されるのに、目的に近づいたかは一度も検査されない状態が起こりうる。スクラムのうちレトロスペクティブだけを実装し、スプリントレビュー（成果物と目的への進捗の検査）を持たない構成が構造的に生む失敗であり、この1問はその代替として置いている。
- **後継（2026-07-21）:** 当初の問いは公開達成で対象消滅したため、**「著者の判断の質が読み取れる形で成果物に残ったか」へ更新**した（§6「運用モードのゴールを『網羅』ではなく…」）。**検査の役割（進捗の番人）は不変で、測る対象だけがゴールに追随した。**

### セッションの出口は分担で設計する

- **決定:** セッション終了・レトロの発火を、**AI が区切り候補を申告し、人間が終了／レトロ／続行を裁定する**分担として定義した。AI の申告根拠は機械的に判定できるものに限る（①タスクが1つ閉じた ②決定・課題の書き戻しが未了 ③文脈が肥大している）。
- **理由:** 再開手順によって**入口**は仕組み化できたが、**出口**は感覚値に委ねられたままだった。ただし終了判断そのものは AI に渡せない——残り時間や集中力は AI に観測できない。逆に**書き戻しの未了は人間の感覚では検知しにくく、AI 側が検知できる領域**であり、次回セッションの立ち上げコストを跳ね上げる。互いに見えないものを補い合う分担にした。
- **申告を「停止条件」にしなかった理由:** 申告のたびに人間の応答を待って作業を止めると、申告そのものが煩わしくなり、やがて無視される。**規律は、守るコストが高いと形骸化する。**

## 3. ツール・記述の設計判断

### Claude Code native に統一する

- **決定:** 本ガイドを Claude Code ネイティブに統一する。制御planeとチーム共通ルールは `CLAUDE.md`、個人の作業原則は `~/.claude/CLAUDE.md`、Skill は `.claude/skills/`（プロジェクト）と `~/.claude/skills/`（個人）に置く。
- **理由:** 単一ツール前提に揃えることで概念とファイルマップが一貫し、別マシンでも clone だけで運用を再現できる。常時ロードされる `CLAUDE.md` に集約し、タスク固有の情報だけを短命なファイルに逃がす構成が、実践上もっとも扱いやすかった。
- **将来の展開について:** 別のエージェントツールへ展開する場合は、再利用可能な方法論（Skill／メモリの概念）とツール別のマッピングを分けて扱う。本ガイドの規律の大半は前者に属する。

### 仕様駆動は「やるか否か」ではなく「どの重量で回すか」

- **決定:** 仕様駆動を重量の選択問題と位置づけ、主軸は軽量運用（仕様 Markdown ＋ Agent Skills）とする。Spec Kit のフルフローは中〜大規模・多論点の変更に限定する。
- **理由:** 対象タスクの多くは要件が明確で小〜中規模であり、軽量運用で完結した。Spec Kit は Constitution 設計とフェーズ運用に初期コストがかかり、軽量なタスクでは投資対効果が出にくい。
- **誤読の防止:** これは**仕様駆動そのものの否定ではない。** What を先に固定し受入基準（AC）を添える規律は常に維持し、重量級の実現手段を選ばなかっただけである。むしろモデルが強力になるほど「雑に投げて丸ごと任せる」誘因が増す一方、**もっともらしい出力ほど反証が難しくなる**ため、What と AC を固定する規律の価値は上がる。

### 記述規約：削減の主レバーは「英語化」ではなく「冗長削減」

- **決定:** 地の文は日本語（可読性優先）、定着した語のみ英語表記とし、トークン削減は冗長削減に集約する。
- **理由:** 日本語はトークン密度が高いが、削減の主レバーは言語選択ではなく簡潔さと構造である。カタカナで定着した語を英語綴りに戻すのは、文字数が増えて可読性が下がり、得るものが小さい。

### 略語は一覧に集約し、入口と主題箇所でのみスペルアウトする

- **決定:** 専門外の読者に伝わらない略語は `01`「用語凡例」の略語一覧に集約し、入口となる `README.md` と、その略語が主題となる箇所でのみ初出スペルアウトする。`commit` / `push` / `API` 等の定着した技術用語は対象外とする。
- **理由:** `01`〜`06` は拾い読みされるため初出スペルアウトのみでは毎回の併記が冗長になり、一覧のみでは入口で読者が脱落する。両者に役割を分担させた。定着技術用語を訳すのは可読性原則に反し、開発者にはむしろ読みにくい。

### 「Brownfield / Greenfield」を本文から削除する

- **理由:** 日本語の開発現場では認知が一様でなく、調べると大規模基幹システム刷新の移行コンサルティング文脈での使用が目立つ。本ガイドの射程（個人〜チーム規模の AI 協働）と読者の連想が噛み合わない。当初は「外部文献参照のための対応語」として注記を残す判断をしたが、**本ガイドはその外部文献を一つも引用していない**ため、参照先のない対応表は読者に文脈を足すだけで何も返さないと判断し、削除に改めた。
- **補足:** `[Judgment]` 両語は英語圏のソフトウェア工学では広く使われており、語自体が誤りというわけではない。**削除の根拠は語の正しさではなく、この文書における有用性の欠如**である。

### 位置づけを「ナレッジベース」から「実践ガイド」へ変更する

- **決定:** 自己定義を「AI駆動開発 実践ガイド」（副題「アジャイル/スクラムの規律を、AIとの協働に適用する」）とする。
- **理由:** 中身の主題はツールの使い方ではなく**進め方の規律**であり、「ナレッジベース」は読者に Tips 集を期待させて最大の差別化点を隠す。なお「スクラムガイド」への改称は見送った——同名の固有文書と衝突し、「スクラムを教える文書」と誤読されるため。本ガイドはスクラムを教えるのではなく、**その規律を AI 駆動開発に適用している。**

## 4. 誤りの訂正記録（規律が実際に機能した例）

本ガイドの規律は、**自身の記述に対して実行して誤りを検出した**実績がある。ここには代表的な訂正だけを残す（件数は書かない——**件数は増えるたびに腐る**）。

### 裏取り済みの `[Fact]` も、読み違えれば誤る

- **訂正:** サブエージェントの `skills` フィールドについて「明示列挙した Skill **のみ利用できる**」と書いていたが誤り。`skills` が制御するのは「起動時にコンテキストへプリロードするか」であって「利用できるか」ではなく、列挙しなかった Skill も呼び出せる。**アクセス制限ではなくコンテキスト最適化のスイッチ**である。
- **なぜ誤ったか:** この記述には公式ドキュメントで裏取りをして `[Fact]` を付けていた。それでも誤ったのは、出典の「プリロードされない」を「アクセスできない」と**読み違えた**ためである。
- **教訓:** **確度ラベルは「出典を見た」ことの証明であって、「正しく読めた」ことの証明ではない。** 裏取りの有無とは別に、出典の主張と本文の主張が一致しているかを照合する必要がある。この誤りは、書いた直後のレビューではなく、時間を置いた再点検（Check Action）で検出された。

### 引用を自己の実証として提示していた

- **訂正:** ある章で、外部記事が報告しているトークン削減率を**自身の実測であるかのように**記述していた。出典表示を追加し、著者自身の実測ではない旨を明記し、原典に存在しない数値（丸めの過程で生じたもの）も原典どおりに修正した。
- **なぜ問題か:** ①**確度ラベルを説くガイドが、引用を自己の実証として提示していた**——規律の自己適用の失敗であり、読者が原典に到達したときにガイド全体の信頼性を損なう。②本ガイドは表示義務のあるライセンスを掲げており、出典表示を求める側が出典を欠いていた。
- **教訓:** **確度ラベルは「それが自分の観測である」ことの証明ではない。** 実証ケースを書くときは、出自（自己の実測／引用／一般知識）を最初に明示する。

### 主張の格下げは、格下げした節の外へ波及する

- **経緯:** 契約上の判断から、4つの節を「実証ケース」から「適用例」へ書き換えた。これは語の言い換えではなく**主張の格下げ**である。しかし次のセッションの点検で、**同じ題材を指す別の箇所が「（実証済み）」のまま残っている**ことが判明した。格下げは節の中だけで完結していなかった。
- **なぜ見落としたか:** 残作業の一覧が「実プロジェクト」という**語**を検索対象にしており、格下げの実体である「実証」という語を追っていなかった。**一覧そのものが網羅していないことは、一覧を見ても分からない。**
- **教訓:** **ある記述の確度を下げたら、同じ対象を指す他の記述を語で検索して洗う。** 節単位の編集は、参照・要約・補足として散らばった同一主張を取り残す。**検索の起点は「変更した箇所」ではなく「変更した主張を表す語」に置く。**
- **射程の拡張（後日の再発により追記）: 波及するのは「格下げ」だけではない。「解決」「完了」も同じように波及する。**
  - **再発の実例:** ある作業を「🔴 未解決のブロッカー」として計画に書いた。その後、実測して解決済みだと確認したが、**ブロッカーとして書いた箇所を更新しないまま残した**。次に読む人（次セッションの自分を含む）は、不要な作業から始めることになる。
  - **前の実例との違い:** 前者は「確認しなかった」ための誤りだが、**後者は確認したのに、その結果を主張の置き場所へ戻さなかった**。**検証と反映は別の作業であり、前者を済ませても後者は自動的には終わらない。**
  - **とくに壊れやすい記述:** **状態を表すもの**——`🔴 ブロッカー` / `未着手` / `未確認` / `要対応`。これらは**書いた時点では正しく、状況が変われば黙って偽になる**。内容についての主張と違い、**誤りが「古くなる」形で現れるため、読み返しても違和感が出ない。**
  - **人間が並行して動いている項目はとくに危険:** 認証・環境構築・ファイルの移動などは、AI の知らないところで解決されうる（`BACKLOG.md`「人間が並行して環境を変えると、AI は古い前提のまま判断を続ける」）。

### 実行対象のない規約が、制御planeの中で無言のまま生き残っていた

- **訂正:** チーム共通ルールに「変更を完了とする前に、linter エラーの確認とテスト実行で検証する」という行が残っていた。**このリポジトリにはコードもテストも lint 設定も無く、実行対象が1つも存在しない。** さらに悪いことに、同じ `CLAUDE.md` の中で「ナレッジ本文に自動の品質ゲートは無い」と明記しており、**制御planeが自分自身と矛盾していた。**
- **なぜ長く残ったか:** 初期化時に生成された定型文で、**一度も実行されなかったがゆえに、誰にも矛盾を突きつけなかった。** 破れた規約は失敗として現れるが、**実行対象のない規約は何も起こさないため、検査を素通りする。**
- **教訓:** **「守られていない規約」より「実行しようがない規約」のほうが見つけにくい。** 規約を書くときは、それが**何に対して実行されるのか**を同時に確定する。対象を名指せない規約は、書いた時点で死んでいる。
- **置き換えた内容:** 実行対象が実在する検証——「その変更が壊した参照だけを確認する」——へ差し替えた。**スコープ限定（全体整合性は通さない）を注釈ではなく規約本体に埋め込んでいる**。注釈は引用されるときに落ちるため。

### 常時ロードされている規約が、発火せずに素通りした

- **何が起きたか:** 「対話では役割名ではなく実体名で呼ぶ」という規約が `CLAUDE.md` に明文で存在したにもかかわらず、AI はセッションを通じて役割名を使い続けた。**文脈から失われたのではない**——`CLAUDE.md` は毎セッション自動ロードされ、AI はセッション冒頭にこれを読んでいた。**読んだうえで発火しなかった。**
- **原因1（WHEN の欠落）:** 規約は **WHAT（実体名で呼べ）だけを書き、WHEN（いつ発火するか）を書いていなかった。** これは改善 Action の改訂で一度直したのと同じ欠陥であり、`03` が Skill の `description` について言う「WHAT だけでなく WHEN を書く」の**自己適用漏れの2件目**である。
- **原因2（置き場所）:** その一文は「記憶モデル」節の、**ファイルの置き場所を説明する長い引用ブロックの末尾**にあった。周囲はすべて「どのファイルをどこに置くか」の話で、**対話の作法だけが1文まぎれていた**。**読まれる文脈と発火すべき文脈が一致していない規約は、読まれても効かない。**
- **処置:** チーム共通ルールへ独立した1行として移し、発火条件（本文を書く作業の最中に、本文の語彙が対話へ漏れる）を明記した。**新しい追跡項目は作っていない**——欠陥は規約の内容ではなく配置であり、置き直すことが対処そのものだったため。
- **教訓:** **「常時ロードしているから守られる」は成立しない。** ロードは前提条件であって発火条件ではない。前項が「実行対象のない規約」を扱ったのに対し、こちらは**実行対象はあるが、実行の合図が無い規約**である。**規約を書くときは、何に対して実行されるか（対象）と、いつ実行されるか（合図）の両方を確定する。**

### 情緒的な文脈では、確度の検査が緩む

- **訂正:** 労いに応じる文脈で、AI が「**間違えたのは私で、止めたのは全部あなただ**」と述べた。**この全称命題は偽である。** 実測すると、AI 側が検出した誤り（前セッションの再点検で3件）と、指摘される前の自己申告（公開ゲートの例外を通そうとした件）が記録に残っている。**分担を実態より単純化して、一方に寄せた。**
- **見落とされやすい理由:** 誤った断定が**自己卑下の形**をとっていた。「私が悪い・相手が正しい」という配分は、**受け取る側が反論しにくい**——訂正すると相手を否定する形になるためである。**謙遜は、検査を通りやすい断定である。**
- **構造的な問題:** 確度ラベルの規律は、**分析・レビュー・設計といった文脈を想定して設計されていた。** 労い・振り返り・感謝といった register は射程に入っていない。しかし**そこでも事実についての主張は行われる**——しかも、その register では検査そのものが場違いに見える。
- **なぜ実害があるか:** この種の要約は**印象に残りやすく、後から引用されやすい**。「AI は間違え、人間が止める」という単純な図式は、分担設計（`01` §3.6）の記述として流通しうるが、**実態はもっと入り組んでいる**——AI 側にも検出できる領域があり、それを前提に仕組み（時間差の再点検）が組まれている。**図式が単純化されると、その仕組みを持つ理由が消える。**
- **`[Fact]` この訂正を検出したのは AI 自身である。** 著者からの指摘ではなく、**同じ発言を正本に残すかどうかを検討する過程で発見された**。「書き残す価値があるか」を問う行為が検査として働いた——**公開を前提に書くと、検査の基準が上がる。**
- **教訓:** **確度の検査を「分析しているとき」に限定しない。** 事実についての主張は、どの register でも主張である。とくに**分担・貢献・原因の配分を語るとき**は、配分そのものが検証対象になる。

## 5. 公開に関する判断

### ライセンスを分離する（散文は CC BY 4.0、コード断片は MIT）

- **決定:** 散文・方法論は **CC BY 4.0**、コード断片・設定例・テンプレート・SKILL.md 雛形は **MIT** とし、両者の分離を `LICENSE` と `README.md` に明記する。
- **理由:** ①本ガイドは商品ではなく能力の証明であり、複製を制限しても守るものがない一方、流通だけが減る。②**氏名表示は権利ではなく導線**であるため CC0 は採らない。③ NC（非営利限定）は最も届けたい読者である企業のチームを締め出す——業務利用が商業利用に該当しうる。④ SA（継承）は企業法務が忌避し、決裁者層での流通が止まる。⑤ テンプレートは**コピーされる前提**であり、氏名表示義務は守られない——**守るコストが高い規律は形骸化する**という、本ガイド自身が繰り返し確認した原則がここにも当てはまる。
- **追記（2026-07-20・収録の非対称を解消）:** `LICENSE` は MIT を全文で収録する一方、CC BY 4.0 は要約（deed）のみだった。**この非対称は設計判断ではなく書き分けの取りこぼしだった**——本エントリに要約収録についての記述が無かったことで確認できた。**全文の収録ではなく、要約の直前に「これは要約であり、適用されるのは URL の全文である」と明記する**形で解消した。理由は、CC 自身が deed に同じ注記を置いており、**要約が条文を代替しないことが明示されていれば厳密さは揃う**ため。約7,500語の legalcode を収録すると、`LICENSE` を開いた読者が最初に見る情報量が増え、**分離の構造そのものが読みにくくなる**。
- **補足（ライセンスの限界）:** 著作権が保護するのは表現であって方法論ではない。**読んで自分の言葉で書き直す行為は SA でも止められない。** 実効的な防御は「先に公開した記録」だけであり、これは公開を遅らせるほど失われる。

### 個人用ツールの設定ファイルは公開物に同梱しない

- **決定:** 著者個人が使う日本語 lint の設定ファイルと依存関係を、公開リポジトリから外す。設定の実体は、対象（プロンプトの下書き）と同じ非公開リポジトリ側に置く。
- **理由1（誤読）:** **公開物に lint 設定があると、読者は「本文が lint されている」と読む。** 本ガイドの本文に自動の品質ゲートは無く、品質は人間の通読とレビューで担保している。**リポジトリに設定ファイルがあることは、それが品質ゲートである証拠にはならない**——にもかかわらず、設定ファイルの存在自体がそう主張してしまう。実際、この位置づけを誤って文書化しかけ、撤回した経緯がある。
- **理由2（実効性）:** **設定は対象と同じ場所に無いと効かない。** 下書きバッファを別リポジトリへ移した時点で、公開側に残った設定は動作しない。**動かない設定を配布物に残すのは、読者に再現しない手順を渡すことと同じ。**
- **副次的な効果:** 依存関係のロックファイル（3,600行超）が公開物から消えた。**コードを含まない文書リポジトリで最大のファイルが、本文と無関係な個人ツールのものだった。**
- **知見そのものは捨てない:** 「プロンプトを下書きで推敲する」「日英混在の技術文書では日本語 lint のルールを大幅に絞る必要がある」は有用な実践知として `BACKLOG.md` に記録し、本文への追記候補とした。**設定ファイルを同梱することは、知見を伝える手段として弱い**——読者は設定を見ても、なぜそのルールを切ったのかを読み取れない。**伝えるべきは設定ではなく判断である。**

### 公開の条件を「安全性のみ」に固定する

- **決定:** 公開のゲートを安全性の確認だけに限定する。**内容品質はゲートにしない。**
- **理由:** 「全ての発展課題の完了と全章の通読」を実質のゲートにしていたが、**通読は必ず新しい課題を生むため収束しない。** とくに「正本に書いた主張の再点検」は、書き続ける以上いつまでも対象が尽きず、**検出が続くこと自体が「まだ公開できない」根拠として機能し続ける**状態になっていた。ブラッシュアップは公開後に継続する。
- **例外の基準（後日追記）:** **制御plane（`CLAUDE.md`）の自己矛盾・実行対象のない規約は、公開ブロッカーとして扱う。** 本文（`01`〜`06`）の記述誤りは公開後に回す。
  - **線を引く軸:** 「毎セッション自動ロードされ、以後の作業の挙動を変えるか」。制御planeの欠陥は文書の誤りではなく**設定のバグ**であり、放置すると全ての後続セッションに効き続ける。
  - ⚠️ **この例外を先に言語化した理由:** 例外の基準が無いまま「重要だから今直す」を1件通すと、**次も通る**。ゲートは明示的に破られるのではなく、**個別に正当化された例外の累積で溶ける。** 却下できる基準を持つことが、ゲートを維持する実体である。

### バージョンは git tag で表現し、初版を `v1.0.0` とする

- **決定:** リリースは `vX.Y.Z` タグとタグごとの GitHub Release で表現し、**バージョン番号をファイルに持たない**。初版は `v1.0.0` とする。
- **ファイルに持たない理由:** **更新規則のない番号は黙って偽になる。** メタデータファイルの `version` フィールドは、更新を思い出す仕組みが無ければ実態から乖離し、しかも乖離したことが読んでも分からない。これは行番号ベースの索引を廃止したのと同じ理由である（「同じ情報を2箇所に置かない」）。
- **`0.x` を採らなかった理由:** 「全体整合性が未確認」「Skill が未作成」「実プロジェクトへの適用が未着手」を根拠に `v0.5.0` とする案を検討し、退けた。
  - **① 整合性を根拠にすると、外したゲートが別の扉から戻る。** 全体整合性は「公開の条件を安全性のみに固定する」で明示的にゲートから外したもので、版数の条件に置き直せばゲートとして復活する。
  - **② 成果物の外にあるものを完成度に数えない。** Skill 化はロードマップ項目であって、ドキュメントとしての本ガイドの欠落ではない。**この成果物の完成度が、この成果物の外にあるもので決まる**構成になる。
  - **③ `0.x` は「1.0 への経路」を主張する番号である。** その条件を書けないなら、それは削除したばかりの「更新規則のない番号」に戻る。加えて本ガイドには**卒業しない検査項目が居座る**という反復パターンがあり、条件なき `0.x` は永久に `0.x` になる側に賭けることになる。
- **未検証であることは、版数ではなく本文が担う `[Judgment]`:** 本ガイドの過大申告への対処は一貫して**主張ごとの精度**だった（「実証」→「適用」の格下げは節単位の書き換えである）。一律の `0.x` は**完成している部分まで下げ、しかもどこが未検証かは言わない**——精度が落ちる方向の謙遜になる。確度ラベル・`BACKLOG.md`・README の「短時間で中身を確かめたい方へ」の3経路が、既に未検証の所在を名指しで示している。

### 公開後の git 履歴は squash してよい（履歴を記録媒体に数えないことの帰結）

- **決定:** 公開直後の残務コミット4件を初版コミットへ squash し、force push した。
- **根拠:** 本ガイドは **git 履歴を文脈復元の手段に数えない**と決めている（「単一の再開用メモを廃止し、記憶モデルを役割で分ける」）。潰したコミットの内容は全て `.md` 側に正本がある——**履歴は一次情報ではなく複製**であるため、失われるものが無い。加えて4件は公開の1時間以内に起きた公開作業の残務であり、読者にとっての開発史ではない。
- ⚠️ **squash は履歴の削除ではない `[Fact]`:** 到達不能になった旧コミットは GitHub 側に残り、SHA を知っていれば参照できる。**機微情報を含む履歴に対して squash を対処として用いてはならない**——公開時に既存リポジトリへの force push を避け、新規リポジトリを作った判断はこの事実に基づいている。両者を混同すると、その判断が無効になる。

### ソロ公開リポジトリの main は「PR 必須」より「force push/削除禁止」で守る

- **決定（2026-07-22）:** 公開リポジトリの main 保護を、**PR 必須＋承認（approvals）ではなく force push 禁止・ブランチ削除禁止のみ**とする。直 push は維持する。
- **理由 `[Fact]`:** **外部の非 collaborator は最初から main に push できない**（write 権限は collaborator のみ・外部は fork＋PR しかできない）。したがって「他人が main を壊す」脅威は、保護を入れる前から存在しない。実際に守るべきは**作者自身の事故（force push・ブランチ削除）**であり、それには破壊防止だけで足りる。
- **退けた設定と、その失敗 `[Fact]`:** 当初「Require a pull request＋Required approvals 1＋Require approval of the most recent reviewable push（push 本人以外の承認必須）」を設定した。**GitHub は自分の PR を自分で承認できない**ため、**承認できる他者がいないソロ作者は自分の変更を永久にマージできなくなる**——実際に PR #1 でこのデッドロックが発生し、設定を戻した。
- **一般化:** **保護は多いほど良いのではなく、脅威モデルに合わせる。** 脅威（外部からの破壊）が実在しない対象に、実在しないことを確かめずに重い規律（PR＋承認）を課すと、規律が守る相手を間違えたまま**運用側（ここでは作者自身）を縛る**。これは本ガイドが繰り返す「守るコストが高い規律は形骸化する」「機微は設定でなく構造で守る」と同じ系譜にある——**外部が push 不可という構造が、保護設定より先に効いている。**

### 運用モードのゴールを「網羅」ではなく「著者の判断が読み取れる証拠＋方法論の活性化」に置く

- **決定（2026-07-21）:** 公開達成後の運用モードでは、優先順位と成果物検査の軸を、本文の網羅性から次の2つへ移す。①方法論が実際に発火する形になっているか（Skill 化・活性化）②著者の判断（メタ認知・クリティカルシンキング）が読み取れる形で残っているか。
- **立脚点:** 本ガイドの価値は商品としての完全性ではなく**能力の証明**である（「ライセンスを分離する」と同じ立脚点）。読者は二層——**生成AIを学ぶ人間**と、`.md` を読み込む**生成AIツール自体**。後者を想定するなら、**「読める」より「発火する（走る）」ほうが証拠として強い。**
- **帰結1（PBR・2026-07-21）:** 発展課題の優先順位を再編した。**A（Skill 化＝活性化・看板機能）を筆頭**、**T・U（自らの誤り／速度の危うさを検出した記録＝判断が動いている証拠）を上位群**へ。網羅・裏取り系（C・L・B）は相対的に下げた。**項目 I（Check Action をセッション開始に置く設計の再検討）は明示的に却下**——現行設計に実績があり、形に寄せて発火を失うリスクの方が大きい。「暗黙に却下されたが却下と記録されない課題」を作らないための明示処理である。PBR のスコープは**並び替えと削除のみ**とし、中位の相対順は触っていない（詳細化＝全体レビューは D1 が拒否する失敗モード）。
- **帰結2（MT1 の後継）:** 旧 MT1「このセッションで成果物は公開／利用に近づいたか」は、公開という単一ゴールに紐づいた検査であり、公開達成で**対象が消滅**した。後継を **「このセッションは、著者の判断の質（メタ認知・クリティカルシンキング）が読み取れる形で成果物に残ったか」** とし、`CLAUDE.md` のチーム共通ルールへ反映した。
  - **なぜこの形か:** MT1 の役割は「**プロセス改善が進捗の代わりになる**」失敗（K-1）の番人である。この後継は**網羅を増やしても Yes にならず、判断を可視化したときだけ Yes になる**——番人機能を保ったまま、新しいゴールに沿う。
  - **退けた案:** ①**廃止**——K-1 の番人を失う。実害未観測だが、ゴールが明確化した今は廃止より更新が筋。②**「読者にとっての価値が増えたか」**——射程が広く曖昧で**ほぼ常に Yes** になり、番人が緩む。③**「活性化に近づいたか」**——狭すぎて本文追記系のセッションで常に No になる。活性化の観点は"上位に置く基準"として PBR 側で既に効いており、検査の問いは判断の可視化を測るほうが適切。

### 方法論を Skill 化するとき、Skill 自体に falsify 可能な AC を持たせる（自己適用）

- **決定（2026-07-22・`requirements-review` Skill 初版）:** `06` の方法論を SKILL.md 化する際、①**Skill 自体の受入基準（AC）** を観測可能な条件で明記する ②**description で WHAT・HOW の境界を切る**——の2つを規律とした。requirements-review では AC を4つ（対象不十分なら埋めず一文で返す／確定 What に AC 必須／採否可能な出力すべてに一意 ID／未明示は `[Assumption]` で要確認）に固定し、description に「実装・コードは対象外」を明示した。
- **理由（AC 側）:** `06` は「成果物（要件）の AC」は説くが「Skill が機能したと言える AC」は持たない。**「良いレビューをする」は falsify 不能で AC にならない**——本ガイドが説く「What を検査可能な形で固定する」規律を、Skill 自身に適用しないのは自己矛盾である。Skill の WHAT を壁打ちする過程で、この穴（Skill 用の AC の不在）が最も突くべき点だった。
- **理由（description 側）:** Skill の発火はモデルが `description` を意味的照合して自ら起動判断する（`03` §1）。`requirements-review`（WHAT）と `code-review`（HOW）を同じ repo に置くため、境界語を描かないと相互誤発火する。とくに「設計レビュー」はアーキ実装（HOW）にも読めるため「仕様レビュー」に置換し、対象外を明示した。
- **進め方の記録:** SKILL.md を直接書かず、**`06` の方法論で WHAT（目的・トリガー・AC・出力）を先に壁打ちしてから清書**した（BACKLOG 項目 A の手順。方法論を説くファイルを、その方法論自身の手順で作る＝自己適用）。
