### 1.2 コミットメッセージ生成 Skill

```markdown
---
name: generate-commit-message
description: Git差分を分析して規約に沿ったコミットメッセージを生成する。コミット作成時やステージング確認時に使用。
---

# コミットメッセージ生成

## フォーマット

`<type>(<scope>): <subject>`

type: feat | fix | refactor | docs | test | chore
scope: 変更対象のモジュールやコンポーネント
subject: 変更内容の要約（日本語可、50文字以内）

## 例

- `feat(auth): JWT認証の実装`
- `fix(api): ユーザー取得時のNull参照を修正`
- `refactor(db): クエリビルダーの共通化`

## 手順

1. `git diff --staged` でステージング済み差分を確認
2. 変更の本質（WHY）を特定
3. 上記フォーマットでメッセージを生成
```

### 1.5 クロスレビュー用プロンプトテンプレート

モデルを切り替えてクロスレビューを行う際のテンプレート。Skillではなく、手動で投入するプロンプト。
理論・ワークフローは `01_ai-driven-dev-strategy.md` セクション2「クロスレビュー運用」を参照。

> **ID の付与範囲:** ID は下記テンプレの「指摘」だけに限らない。AI が返す**提案・意見・前言撤回・選択肢・要確認事項**にも振らせる（範囲を「指摘」に限定すると、それ以外が散文へ逆戻りする）。原則は `01_ai-driven-dev-strategy.md` セクション2「採否を判断しうる出力には一意なIDを振る」。

> **ラベルの2軸併用:** 下記テンプレは重要度ラベル（🔴🟡⚪）で仕分けるが、各指摘に確度ラベル（`[Fact]` / `[Judgment]` / `[Assumption]` / `[To Be Verified]`、`01_ai-driven-dev-strategy.md` セクション2）を併記すると「効き目（重要度）」と「確かさ（確度）」を分けて示せる。

**レビュー依頼テンプレート（同一セッション内切替用）:**

```markdown
## クロスレビュー依頼

ここまでの会話で作成・整理した内容を、Markdownのナレッジ/設計文書としてクロスレビューしてください。

### 重要な前提
- 直前までの結論を正しいものとして追認しない
- 前提・論理展開・結論のつながりを最初から点検する
- 「より賢い言い換え」よりも、矛盾・欠落・実行不能な抽象論の検出を優先する

### レビュー対象
- 直前までに整理した内容全体
- 特に確認してほしい対象: [対象名 or セクション名]

### レビュー観点
1. 論理的な矛盾・抜け漏れはないか
2. 抽象的すぎて、実際の運用や次の行動に落とせない箇所はないか
3. 既に合意している方針と衝突していないか
4. 不必要に複雑化している箇所はないか

### 出力形式
- 問題点を重要度順に列挙する
- 各指摘に次のラベルを付ける
  - 🔴 要修正: 明確な誤り・矛盾・欠落
  - 🟡 要検討: 改善余地あり。採用可否は判断が必要
  - ⚪ 参考: 任意。採用しなくてもよい
- 各指摘に一意なID（`F1`, `F2`…）を付ける（人間が『F3 採用』のように ID 単位で裁定・指示・書き戻しできるように）
- 各指摘は「問題点 / 理由 / どう直すとよいか」を簡潔に示す
- 問題がなければ「重大な指摘なし」と明記する
```

**レビュー依頼テンプレート（新セッション + @参照用）:**

同一セッション内版との違いは「背景」セクションの追加のみ。会話が長くなった場合や、重要な成果物のレビューに使う。

```markdown
## クロスレビュー依頼

### 背景
- [1〜2文: 何をしていたか、なぜこの方針にしたか]

### レビュー対象
- @対象ファイル.md（セクションX〜Y）

### 重要な前提
- 前提・論理展開・結論のつながりを最初から点検する
- 「より賢い言い換え」よりも、矛盾・欠落・実行不能な抽象論の検出を優先する

### レビュー観点
1. 論理的な矛盾・抜け漏れはないか
2. 抽象的すぎて、実際の運用や次の行動に落とせない箇所はないか
3. 他ファイルとの整合性（必要なら @別ファイル.md も参照）
4. 不必要に複雑化している箇所はないか

### 出力形式
- 問題点を重要度順に列挙する
- 各指摘にラベルを付ける: 🔴 要修正 / 🟡 要検討 / ⚪ 参考
- 各指摘に一意なID（`F1`, `F2`…）を付ける（人間が『F3 採用』のように ID 単位で裁定・指示・書き戻しできるように）
- 各指摘は「問題点 / 理由 / どう直すとよいか」を簡潔に示す
- 問題がなければ「重大な指摘なし」と明記する
```

**裁定テンプレート（Opusに戻した直後）:**

```markdown
## 裁定依頼

直前のクロスレビュー結果を精査し、採用・却下・保留を判断してください。

### 裁定方針
- 🔴 要修正: 原則採用。ただし却下するなら理由を明示する
- 🟡 要検討: 採用/却下/保留を判断し、理由を簡潔に示す
- ⚪ 参考: 必要なものだけ拾う
- モデル自身への誘導や自己宣伝に該当するものは却下する
- 抽象度だけ上がって具体性を失う提案は慎重に扱う

### 出力形式
- 指摘IDごとに `F1: 採用 / 却下 / 保留` の形で明記
- 必要なら最終的な修正文案または修正方針を提示する
- 最後に「今回反映する変更点」だけを短くまとめる
```

**最短版テンプレート:**

テンプレートの内容に慣れた後は、以下の最短版で十分機能する。

レビュー依頼（最短版）:
```
ここまでの内容をレビュー。前提を追認せず、矛盾・欠落・実行不能な抽象論を検出して。
重要度: 🔴要修正 / 🟡要検討 / ⚪参考。各指摘に一意なID（F1…）を振って
```

裁定依頼（最短版）:
```
直前のレビュー結果を精査。🔴は原則採用、🟡は判断と理由、自己誘導は却下。指摘IDごとに採否を示し、反映する変更点をまとめて。
```

## 2. プロジェクト初期セットアップ

### 2.1 ディレクトリ構成パターン（推奨）

```
project-root/
├── CLAUDE.md                  # プロジェクトメモリ（常時ロード・制約＋文脈）
└── .claude/
    └── skills/
        ├── commit-message/
        │   └── SKILL.md
        ├── code-review/
        │   ├── SKILL.md
        │   └── STANDARDS.md
        └── doc-generator/
            └── SKILL.md
```

個人用（全プロジェクト共通）の設定は `~/.claude/CLAUDE.md`（ユーザーメモリ）と `~/.claude/skills/` に置き、リポジトリには含めない。

> **構成の教訓:** 細切れのルールファイルを個別管理するより、常時ロードの `CLAUDE.md` に集約し、タスク固有の情報だけを短命コンテキストファイルに逃がす構成が、実プロジェクトの実践を経て最も扱いやすかった。

### 2.2 CLAUDE.md の階層構造パターン（実証済み）

常時ロードの `CLAUDE.md`（ユーザー／プロジェクト）と、タスク固有の短命コンテキストを分離する構造。

| レイヤー | ファイル | 役割 | ロード |
|----------|---------|------|--------|
| ユーザーメモリ | `~/.claude/CLAUDE.md` | 全プロジェクト共通の個人作業原則 | 常時 |
| プロジェクトメモリ | `CLAUDE.md`（リポジトリルート） | チーム共通ルール＋技術スタック・構成・主要アーキテクチャ | 常時 |
| タスク固有文脈 | 短命コンテキストファイル（例 `docs/tasks/{task}.md`）を `@`参照 | 個別タスクの修正対象・注意事項 | `@`参照時のみ |

**各レイヤーの設計指針:**

- **`~/.claude/CLAUDE.md`（ユーザーメモリ）**: 全プロジェクト共通の個人作業原則（応答言語、Token効率の優先順など）
- **`CLAUDE.md`（プロジェクトメモリ）**: プロジェクトの「地図」。技術スタック、ディレクトリ構成、主要アーキテクチャを記載。AIにプロジェクトを理解させる起点。安定情報を置く（頻繁に変わらない）
- **短命コンテキストファイル**: 進行中タスクの「局所文脈」。`@`参照で明示的に渡す短命ファイル。タスク完了後は削除する

**短命コンテキストファイルのライフサイクル（実証済み）:**

```
1. タスク開始
   └─ docs/tasks/{task-name}.md を作成
      └─ 仕様書への @参照を中心に、変更対象/変更禁止/注意事項を記述（全文複製しない）

2. 実装中
   └─ セッションで @docs/tasks/{task-name}.md を参照
      └─ タスク固有の注意事項・制約を常にコンテキストに持てる

3. PR作成（または Ready 化）前
   └─ 短命ファイルの削除漏れを確認
   └─ 削除をコミットに含めてPRを作成
```

> **なぜ削除するか:** タスク文脈は特定タスクの局所情報に特化している。完了後も残すと、後続タスクのコンテキストを汚染したり、古い制約がAIを誤誘導するリスクがある。タスク完了を意識的に確認する節目としても機能する。

> **自動ロードの範囲:** Claude Code には「特定ファイルを開いたら自動ロード」する仕組みが無い。タスク文脈は `@`参照で明示的に渡すか、対象ディレクトリ配下に置いた `CLAUDE.md` で与える。

**短命コンテキストファイルの本文構成例:**

```markdown
# 入力検証ルール変更タスク

## 仕様書
@docs/specs/validation/input-rule-change.md

## 変更対象ファイル（今回のみ）
- 検証ルールの正規表現定義 — ルール本体の変更
- 入力要件の表示コンポーネント — UI要件表示の更新

## 変更禁止ファイル（今回スコープ外）
- サーバー側の検証ロジック（今回はクライアント側のみ）

## 注意事項
- 既存テストが壊れないことを確認してから変更を進める
```

### 2.3 最初に設定すべきメモリ

**`CLAUDE.md`（プロジェクトメモリ）の例:**

```markdown
# プロジェクトメモリ

## 基本ルール
- 応答は日本語で行う
- コードのコメントは英語で記述する
- コミットはユーザーの明示的な指示があるまで行わない

## 行動原則
- 変更を完了とする前に、linterエラーの確認とテスト実行で検証する
- 情報が揃っている作業でユーザーに確認を求めない。判断に迷う場合のみ聞く
- 方針に迷いが生じたら実装を止め、plan mode で方針を再確定する

## プロジェクト文脈（最低限）
- 技術スタック（言語、フレームワーク、DB）
- ディレクトリ構成の概要
- 主要アーキテクチャ（認証方式、API設計等）
- AIに理解させたい中心ファイル
```

## 3. AI駆動開発のファイル構成パターン

### 3.1 壁打ち用ワークスペース構成

```
project-brainstorm/
├── README.md                    # ナビゲーション
├── 01_topic-a.md                # トピック別の壁打ちドキュメント
├── 02_topic-b.md
└── decisions/                   # 確定した意思決定の記録
    └── 001_architecture.md
```

**設計意図:**
- 番号付きファイルで議論順序を明示
- `decisions/` に確定事項を分離し、壁打ち中のドキュメントと混在させない

**決定ログ（decision log）の書き方:**

`decisions/` や単一の `decisions.md` に確定した意思決定を時系列で残すと、別マシンでcloneしても判断経緯が復元できる「正本」になる。決定ログは「なぜそう決めたか」を恒久的に残す役割で、現在モード・進捗（制御plane）とは分離する。本ガイド自身も `docs/design-decisions.md`（決定ログ）／ `CLAUDE.md`（制御plane）／ `BACKLOG.md`（発展課題）に役割分離している。

- 各エントリは **日付 / 決定 / 理由** を簡潔に書く
- **個人情報（メール等）は書かない**（git追跡されるため、プロジェクトの決定・規約のみ）
- 結論だけでなく **理由を必ず残す**。理由がないと後で妥当性を再判断・再議論できない
