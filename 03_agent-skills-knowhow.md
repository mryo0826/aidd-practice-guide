# 03. Agent Skills設計・運用ノウハウ

## 1. Agent Skillsとは

Claude Code に特定のタスクの実行方法を教えるMarkdownファイル（`SKILL.md`）。
`[Fact]` モデルが各 Skill の `description` を意味的に照合し、関連すると判断したときに**自ら呼び出す**（受動的な「自動検出」ではなくモデル駆動の起動判断。`description` の精度が発火の可否を左右する ＝ セクション3.1）。

### 保存場所と使い分け

| 種別 | パス | スコープ | 用途 |
|------|------|----------|------|
| 個人用 | `~/.claude/skills/skill-name/` | 全プロジェクト共通 | 個人のワークフロー標準化 |
| プロジェクト用 | `.claude/skills/skill-name/` | リポジトリ内で共有 | チーム共通の開発規約・手順 |

### 判断基準

「別のプロジェクトでも使えるか？」で判断する。

| 質問 | 答え | 配置先 |
|------|------|--------|
| このSkillは別プロジェクトでもそのまま使えるか？ | Yes | 個人用（`~/.claude/skills/`） |
| このSkillはこのプロジェクト固有の手順・規約を含むか？ | Yes | プロジェクト用（`.claude/skills/`） |

## 2. Skills vs メモリ（CLAUDE.md）の使い分け

| 観点 | Skills (`.claude/skills/`) | メモリ（CLAUDE.md） |
|------|---------------------------|-------|
| 目的 | 特定タスクの実行手順を教える | コーディング規約・振る舞いの制約を与える |
| 粒度 | タスク単位（「PRレビューの仕方」等） | 常時適用のルール（「常に日本語で応答」等） |
| 発動 | タスクに関連すると判断された時 | 常にコンテキストに含まれる |
| 構造 | ディレクトリ + SKILL.md + 補助ファイル | 下記の階層構造を推奨 |

**メモリ（CLAUDE.md）の推奨階層構造（実証済み）:**

| レイヤー | ファイル | 役割 | 寿命 |
|----------|---------|------|------|
| ユーザーメモリ | `~/.claude/CLAUDE.md` | 全プロジェクト共通の個人作業原則。常時適用 | 恒久 |
| プロジェクトメモリ | `CLAUDE.md`（リポジトリルート） | チーム共通ルール＋技術スタック・文脈。常時適用 | 恒久 |
| タスク固有文脈 | 短命コンテキストファイル（`@`参照 / ディレクトリ別 `CLAUDE.md`） | 個別タスクの修正対象・局所文脈 | **短命（タスク完了後に削除）** |

タスク固有の短命コンテキストファイルはタスク開始時に作成し、PR作成前に削除する。完了後も残すと後続タスクのコンテキストを汚染するため、削除はタスク完了の確認節目として扱う。

> **タスク文脈の渡し方:** Claude Code には「特定ファイルを開いたら対応するルールを自動ロード」する仕組みは無い。タスク固有文脈は `@`参照で明示的に渡すか、対象ディレクトリ配下に置いた `CLAUDE.md`（そのディレクトリで作業する間ロードされる）で与える。

> 詳細な設計指針・ライフサイクル・frontmatter例は `05_templates-and-patterns.md` のセクション2.2 を参照。

**教訓・気づきの扱い方（一時メモ → メモリ昇格パターン）:**

開発中に得た教訓や気づきは、以下のフローで段階的に定着させる。専用の `lessons.md` のようなファイルは設けず、既存の仕組みで回す。

```
教訓・気づきが発生
  ├─ 一時的・タスク固有 → 短命コンテキストファイルに記録（短命）
  └─ 繰り返し使いそう
       ├─ 行動原則レベル → CLAUDE.md に1行追加（メモリ昇格）
       └─ 手順・ワークフロー → Skill として整備
```

| 判断 | 置き場 | 寿命 |
|------|--------|------|
| 今回だけ覚えておけばよい | 短命コンテキストファイル | タスク完了まで |
| 2回以上繰り返した原則 | `CLAUDE.md` に短い1行で追加 | 恒久 |
| 手順として定型化できる | Skill（`SKILL.md`） | 恒久 |

> **ポイント:** 教訓をいきなり `CLAUDE.md` に書かない。まず一時メモに置き、繰り返し発生したものだけを昇格させる。これにより `CLAUDE.md` が肥大化するのを防ぐ。

**methodology（方法論）と content（固有文脈）の分離:**

Skillには**再利用可能な方法論**（手順・観点・行動規範）だけを書き、**プロジェクト固有のcontent**（要件・数値・対象名など可変値）はSkillに埋め込まず、別の文脈ファイル（仕様書・`CLAUDE.md` 等）に置く。

| 種別 | 置き場 | 例 |
|------|--------|-----|
| 再利用可能な方法論（How） | Skill（`SKILL.md`） | レビューの観点、壁打ちの進め方、確度ラベルの運用 |
| プロジェクト固有のcontent（What） | 仕様書 / context ファイル | 対象機能の要件、納期・数値、固有の制約 |

> **なぜ分離するか:** 方法論とcontentを混ぜると、Skillが特定プロジェクトに癒着し他プロジェクトで再利用できなくなる。日付・数値など可変値はSkillに入れずcontent側に置くことで、Skillを横断的な資産として保てる。

## 3. 効果的なSKILL.mdの書き方

### 3.1 descriptionが最重要

エージェントがSkillを使うかどうかは `description` で決まる。

**良い例:**
```yaml
description: Git差分を分析して説明的なコミットメッセージを生成する。コミットメッセージの作成やステージング変更のレビュー時に使用。
```

**悪い例:**
```yaml
description: Gitのヘルパー
```

ポイント:
- **WHAT**（何をするか）と **WHEN**（いつ使うか）を両方含める
- 三人称で書く（「〜を処理する」「〜時に使用」）
- トリガーとなるキーワードを含める

### 3.2 本文の設計原則

| 原則 | 説明 |
|------|------|
| 簡潔さ最優先 | エージェントは賢い。知っていることを説明しない |
| 500行以内 | SKILL.mdは簡潔に。詳細は別ファイルへ |
| 段階的開示 | 必須情報→SKILL.md、詳細→reference.md |
| 参照は1階層まで | SKILL.md → reference.md は可。reference.md → detail.md は避ける |

### 3.3 設計パターン

| パターン | 用途 | 例 |
|----------|------|-----|
| テンプレート | 出力形式が決まっている | レポート生成、コミットメッセージ |
| ワークフロー | 手順が決まっている | フォーム入力、デプロイ手順 |
| 条件分岐 | 状況によって手順が変わる | ドキュメント修正（新規 vs 既存） |
| フィードバックループ | 品質担保が必要 | コードレビュー、簡易な仕様変更 |

## 4. Skillのディレクトリ構成

```
skill-name/
├── SKILL.md              # 必須：メインの指示書
├── reference.md          # 任意：詳細なリファレンス
├── examples.md           # 任意：使用例
└── scripts/              # 任意：ユーティリティスクリプト
    └── validate.py
```

スクリプトの利点:
- 生成コードより信頼性が高い
- Token節約（コンテキストにコードを含めなくて済む）
- 実行結果の一貫性を担保

## 5. Skillsの粒度・数の目安

| 観点 | 推奨 |
|------|------|
| 粒度 | 1 Skill = 1タスク（「コミット」「レビュー」「テスト実行」等） |
| プロジェクト用の上限 | 7〜8個（多すぎるとdescription競合が増える） |
| 個人用の上限 | 10〜15個（汎用的で競合しにくいがメンテコスト増大） |
| 初期導入 | 2〜3個から始め、Pain Driven で追加する |

粒度が粗い Skill は description が曖昧になり、エージェントがトリガーを外しやすくなる。
迷ったら分割する。
